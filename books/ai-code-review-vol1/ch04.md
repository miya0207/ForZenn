# 第4章: 人間が担うべきレビュー

:::message
**この章で学べること**
- AIでは判断できない「文脈を要するレビュー」の5領域
- 各視点の「良い指摘の例 vs 悪い指摘の例」
- 人間レビューの時間を設計議論に集中させる方法
- 設計パターンの問題（SOLID原則・循環依存）のAI検出手法

## 3.1 視点①：アーキテクチャ適合性

「このコードは動くか」ではなく「このコードはシステム設計と整合しているか」を問う視点。これはAIに不可能な判断だ。

### 実例：レイヤー違反

```python
# ❌ レイヤー違反（ControllerからRepositoryを直接呼ぶ）
# controllers/user_controller.py
class UserController:
    def get_profile(self, user_id: int) -> dict:
        # ControllerがDBに直接アクセスしている
        conn = sqlite3.connect("app.db")
        cursor = conn.execute(
            "SELECT * FROM users WHERE id = ?", (user_id,)
        )
        user = cursor.fetchone()
        return {"id": user[0], "name": user[1]}
```

AIはこのコードを「動作する」と評価するかもしれない。しかし、システムが「Controller → Service → Repository」の3層アーキテクチャで設計されている場合、これは明確な違反だ。

**人間のレビュアーの指摘（適切な例）:**
```
このControllerがDBに直接アクセスしています。
- UserServiceを通してください（既存のUserService.get_by_id()が使えます）
- ControllerにSQL文が入ると、単体テストでDBモックが必要になります
- 今後3箇所でこのパターンが再現すると、全部直す羽目になります
```

### アーキテクチャ設計への質問チェックリスト

```
レビュー時に自問すべき問い:
□ このクラスはどのレイヤーに属するか？そのレイヤーの責任範囲を守っているか？
□ 依存の向きは正しいか？（上位レイヤーが下位レイヤーに依存しているか）
□ このモジュールを別のプロジェクトで再利用できるか？（再利用性）
□ この実装は既存のアーキテクチャ決定（ADR）と矛盾していないか？
□ DIコンテナを使っているなら、依存関係は設定ファイルで管理されているか？
```

## 3.2 視点②：保守性と将来の変更コスト

「今は動く」だけでなく「半年後も変更しやすいか」を評価する。これはコードを書いた文脈と、チームの将来計画を知っていないと判断できない。

### 実例：変更に弱い設計

```python
# このコードの問題は何か？動作は完全に正しい。
class NotificationService:
    def send(self, user_id: int, message: str, channel: str = "email") -> bool:
        if channel == "email":
            return self._send_email(user_id, message)
        elif channel == "slack":
            return self._send_slack(user_id, message)
        elif channel == "push":
            return self._send_push(user_id, message)
        else:
            raise ValueError(f"Unknown channel: {channel}")

    def _send_email(self, user_id: int, message: str) -> bool: ...
    def _send_slack(self, user_id: int, message: str) -> bool: ...
    def _send_push(self, user_id: int, message: str) -> bool: ...
```

AIは「動作する」と言う。しかし人間のレビュアーは：

**悪い指摘（曖昧）:**
```
「if-elif は悪いパターンです。Strategy Patternを使ってください」
```

**良い指摘（文脈付き）:**
```
「来月、SMS通知の追加が予定されていますね（QAで言及がありました）。
今の設計だと、追加のたびにこのif-elifが長くなり、send()をテストするたびに
全チャンネルの動作を検証する必要が出てきます。

戦略パターンで書き直すと：
- channel追加 = 新クラス追加のみ（既存コードを変更しない）
- channel ごとの単体テストが書きやすい

ただし、チャンネルが増える予定がないなら今のままで問題ありません。
どちらが現実的ですか？」
```

**改善後のコード:**

```python
from abc import ABC, abstractmethod

class NotificationChannel(ABC):
    @abstractmethod
    def send(self, user_id: int, message: str) -> bool: ...

class EmailChannel(NotificationChannel):
    def send(self, user_id: int, message: str) -> bool:
        # メール送信実装
        return True

class SlackChannel(NotificationChannel):
    def send(self, user_id: int, message: str) -> bool:
        # Slack送信実装
        return True

class NotificationService:
    def __init__(self, channels: dict[str, NotificationChannel]):
        self._channels = channels

    def send(self, user_id: int, message: str, channel: str = "email") -> bool:
        if channel not in self._channels:
            raise ValueError(f"Unknown channel: {channel}")
        return self._channels[channel].send(user_id, message)

# SMS追加は新クラスを追加するだけ
class SMSChannel(NotificationChannel):
    def send(self, user_id: int, message: str) -> bool:
        # SMS送信実装
        return True

# 使用例
service = NotificationService({
    "email": EmailChannel(),
    "slack": SlackChannel(),
    "sms": SMSChannel(),  # ← 追加はここだけ
})
```

## 3.3 視点③：ビジネスルールの妥当性

コードが「仕様書通り」かどうかの判断。ビジネスドメインの知識なしに判断できない。

### 実例：仕様書の落とし穴

```python
def calculate_shipping_fee(weight_kg: float, destination: str) -> int:
    """
    配送料を計算する。
    - 国内: 重量 × 100円
    - 海外: 重量 × 300円
    """
    if destination == "domestic":
        return int(weight_kg * 100)
    else:
        return int(weight_kg * 300)
```

AIはこのコードを仕様通りと評価する。しかし人間のレビュアーは質問する：

```
【確認事項 — ビジネスルールの観点から】

1. 重量が0kgの場合はどうなりますか？（仮想的に0円になる）
   → 最低配送料の設定は必要ではないか

2. 重量が100kgを超えた場合は通常配送扱いでよいですか？
   → 大型配送業者への委託ルールはないか

3. "destination" が "domestic" でも "international" でもない値
   （例: "overseas", "foreign"）が来た場合、海外扱いになりますが問題ないですか？

4. 離島（沖縄・北海道）は「国内」扱いでよいですか？
   実際には国内でも追加料金が発生するケースがあります。

5. 消費税はどこで計算しますか？この関数の呼び出し元で加算しますか？
   それとも税込みで返すべきですか？
```

これらは「コードの正しさ」ではなく「ビジネスの正しさ」の問いだ。AIは仕様書を読んでいないし、物流部門の担当者との会話も知らない。

### ビジネスルールレビューの問いリスト

```
【ビジネスルールチェックのテンプレート】

境界値の確認:
□ 入力値が0、負数、最大値の場合の動作は仕様通りか？
□ 空文字列、空配列、Noneが来たとき、ユーザーにとって適切な応答をするか？

例外ケースの確認:
□ 特定のユーザー種別（管理者、プレミアム、無効化済み）で動作は変わるか？
□ タイムゾーン・言語設定で結果が変わる処理はないか？

税金・料金計算:
□ 消費税の計算はどこで行うか（内税・外税）？
□ 割引が複数適用される場合の優先順位は？

データの整合性:
□ この処理が失敗した後、関連するデータは一貫した状態を保つか？
□ 並行して同じ処理が走った場合（二重送信など）の動作は？
```

## 3.4 視点④：テスタビリティ

「テストが書けるか」は設計の質の指標だ。テストの書きにくいコードは、多くの場合、設計上の問題を抱えている。

### テスタビリティを下げる典型パターン

```python
# ❌ テストが書きにくいコード

import datetime
import requests

class ReportGenerator:
    def generate_daily_report(self) -> dict:
        # 問題1: datetime.now() をテストできない
        today = datetime.datetime.now().date()

        # 問題2: 外部APIが必要（テスト環境で動かない）
        response = requests.get(
            f"https://api.example.com/sales?date={today}"
        )
        sales_data = response.json()

        # 問題3: グローバル設定に依存（テストで状態汚染）
        config = global_config.get_report_settings()

        return {
            "date": str(today),
            "total_sales": sum(d["amount"] for d in sales_data),
            "currency": config["currency"],
        }
```

このコードをテストするには、本物のAPIが動いている必要があり、テスト実行日によって結果が変わり、グローバルな設定が特定の値になっている必要がある。

**人間のレビュアーのコメント（テスタビリティ観点）:**

```
このコードにユニットテストを書くのが難しい構造になっています。

問題:
1. datetime.now() が関数内でハードコード → 日付を指定してテストできない
2. requests.get() が内部で呼ばれている → APIモックが必要
3. global_config への直接依存 → テスト時の設定が予測不能

改善後:
```

```python
# ✅ テストが書きやすいコード（依存性を外から注入）
from datetime import date
from typing import Protocol

class SalesDataFetcher(Protocol):
    def fetch(self, target_date: date) -> list[dict]: ...

class ReportConfig(Protocol):
    currency: str

class ReportGenerator:
    def __init__(
        self,
        fetcher: SalesDataFetcher,
        config: ReportConfig,
    ) -> None:
        self._fetcher = fetcher
        self._config = config

    def generate_daily_report(self, target_date: date | None = None) -> dict:
        # date を引数で受け取れる（デフォルトは今日）
        report_date = target_date or date.today()
        sales_data = self._fetcher.fetch(report_date)

        return {
            "date": str(report_date),
            "total_sales": sum(d["amount"] for d in sales_data),
            "currency": self._config.currency,
        }

# テスト例
def test_generate_daily_report():
    # モックを差し込む
    class MockFetcher:
        def fetch(self, _date):
            return [{"amount": 1000}, {"amount": 2000}]

    class MockConfig:
        currency = "JPY"

    generator = ReportGenerator(MockFetcher(), MockConfig())
    report = generator.generate_daily_report(date(2024, 1, 15))  # 日付を固定

    assert report["date"] == "2024-01-15"
    assert report["total_sales"] == 3000
    assert report["currency"] == "JPY"
```

## 3.5 視点⑤：チームの慣習・コードベースとの整合性

チームには明文化されていない「暗黙のルール」が多数存在する。新しいコードがそのルールを守っているかは、チームの文脈を知っている人間にしか判断できない。

### 暗黙のルールの例

```python
# ── チームの暗黙ルール例 ──────────────────────────────────
# 1. エラーハンドリングは全て service 層で行い、上位には例外を伝播させない
# 2. 外部API呼び出しは全て api_client/ ディレクトリに集約する
# 3. ログはstructured logging（JSON形式）で書く
# 4. データベースのモデルを直接 Controller から参照しない
# 5. 設定値は全て config.py 経由で取得する（os.environ を直接使わない）

# ❌ チームのルールを破っているコード（AIは気づかない）
class UserController:
    def create_user(self, data: dict) -> dict:
        import os
        api_key = os.environ["EXTERNAL_API_KEY"]  # ルール5違反

        try:
            user = User(**data)
            db.save(user)
            # 外部API呼び出しをControllerに直接書いている（ルール2違反）
            response = requests.post(
                "https://analytics.example.com/events",
                json={"event": "user_created", "user_id": user.id},
                headers={"Authorization": f"Bearer {api_key}"}
            )
            return {"id": user.id, "status": "created"}
        except Exception as e:
            # Controllerでエラーハンドリング（ルール1違反）
            logger.error(f"User creation failed: {e}")
            return {"error": str(e)}
```

AIはこのコードの「動作」は正しいと判断する。しかしチームのルールを知っている人間は全ての違反を指摘できる。

### レビューコメントの書き方（人間としての質を上げる）

良いレビューコメントは「何が問題か」だけでなく「なぜ問題か」と「どう直すか」を含む。

```
❌ 悪いコメント:
「os.environ を直接使わないでください」

✅ 良いコメント:
「os.environ の直接参照は config.py 経由に変えてください（チームルール#5）。
理由: 環境変数の一覧が config.py に集約されることで、
・必要な環境変数が一目でわかる
・デフォルト値や型変換が一箇所に集まる
・テスト時の mock が簡単になる

変更例:
  from config import get_config
  config = get_config()
  api_key = config.external_api_key
```

## 実務Tips

**Tip 1: チームのレビュー視点をドキュメント化する**
「このチームが重視するレビュー観点」をREADMEやWikiに書いておく。新メンバーのオンボーディングにも使え、AIプロンプトにも追加できる。

**Tip 2: 「後で困ること」を言語化する習慣をつける**
「将来の変更コスト」を具体的に予測して伝える。「設計が悪い」ではなく「3ヶ月後にタグ検索機能を追加するとき、このスキーマだと全テーブルのマイグレーションが必要になります」と伝える。

**Tip 3: レビューで「なぜ」を聞く**
「なぜここでこの実装を選んだのですか？」という問いが、実はビジネスルールの抜け漏れや設計上の誤解を発見する最善の方法だ。

## 4.6 AIが間違えた実際の事例

これは重要だ。AIを盲信してはいけない理由を、実際に起きた事例で示す。

### 事例①：スレッドセーフを誤って「安全」と判断

```python
# AIが「問題なし」と判断したコード
class RequestCounter:
    count = 0  # クラス変数

    def increment(self):
        self.count += 1  # ← 複数スレッドから呼ばれると競合状態が発生
```

Claudeのコメント（実際の出力）:
```
このコードは基本的な整数カウンターの実装で、機能的に正しいです。
count はクラス変数として全インスタンスで共有されます。
```

**実際の問題:** `self.count += 1` はPythonでは非アトミック操作（GILがあっても`__iadd__`の途中で割り込みが入る場合がある）。マルチスレッド環境でカウントが狂う。

**正しい実装:**
```python
import threading

class RequestCounter:
    def __init__(self):
        self._count = 0
        self._lock = threading.Lock()

    def increment(self):
        with self._lock:
            self._count += 1

    @property
    def count(self):
        return self._count
```

**教訓:** AIはマルチスレッドの動的な問題（実行時の競合状態）を静的解析だけでは完全に把握できない。並行性に関わるコードは人間が必ずレビューする。

---

### 事例②：廃止予定APIを推奨

```python
# AIが提案したコード（古い知識に基づく）
import ssl
context = ssl.SSLContext(ssl.PROTOCOL_SSLv23)  # 推奨されなくなったAPI
```

AIは訓練データのカットオフ時点の知識しか持たない。Python・ライブラリのアップデートで推奨が変わった後も、古いAPIを「正しい」と判断することがある。

**チェック方法:**
```python
# AIの提案を受けたら、まず公式ドキュメントで確認
import ssl
context = ssl.create_default_context()  # 現在の推奨
```

### 事例③：ビジネスルールを「バグ」と誤認識

```python
def calculate_bonus(salary: float, performance: str) -> float:
    """ボーナス計算: 業績S=50%, A=30%, B=10%"""
    rates = {"S": 0.5, "A": 0.3, "B": 0.1}
    return salary * rates.get(performance, 0.0)
```

Claudeのコメント（誤った指摘）:
```
🟡 警告: performance が S/A/B 以外の場合、ボーナスが0になります。
デフォルト値に別の処理を検討してください。
```

**実際:** `performance` が `S/A/B` 以外（評価不能・休職中など）の場合にボーナス0は正しいビジネスルール。AIはドメイン知識を知らないため、正しい仕様を「バグ」と誤認した。

### AIの誤りに対処する原則

```python
# AIレビュー結果を扱うガイドライン
HUMAN_OVERRIDE_RULES = {
    "並行性・スレッドセーフティ": "必ず人間が最終確認",
    "ライブラリの最新推奨": "公式ドキュメントで裏付け確認",
    "ビジネスルールへの指摘": "仕様書・PO確認後に判断",
    "セキュリティ脆弱性の否定": "AIが安全と言っても追加確認",
    "パフォーマンスのベンチマーク": "実測データで検証",
}
```

## まとめ

人間のレビュアーが担う5つの視点：

1. **アーキテクチャ適合性** — レイヤー構造・依存方向がシステム設計と整合しているか
2. **保守性** — 半年後に変更しやすい設計になっているか。変更コストを見積もれるか
3. **ビジネスルールの妥当性** — 境界値・例外ケース・ビジネスの現実を考慮しているか
4. **テスタビリティ** — テストが書けない設計は設計上の問題のシグナルである
5. **チームの慣習との整合性** — 明文化されていないルールを守っているか

そして最も重要な原則：**AIが「問題なし」と言っても、最終判断は常に人間が行う。**

AIが間違える領域（並行性・最新ライブラリ・ドメインルール）を理解し、人間のレビューと組み合わせることで初めてAIコードレビューは機能する。
