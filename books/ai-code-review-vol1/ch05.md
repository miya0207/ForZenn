# 第5章: プロンプト設計 — レビュー品質を決める核心

:::message
**この章で学べること**
- 「バグを探して」が失敗する理由と、失敗パターン3つの解剖
- Role・Task・Format・Constraint の4要素でプロンプトを構造化する方法
- A/Bテストで判明した「効くプロンプト vs 効かないプロンプト」の具体的な差異
:::

## 5.1 「バグを探して」が使えない理由

多くの開発者が最初に試すプロンプトはこれだ。

```
以下のコードのバグを探してください。
```

これが機能しない理由は3つある。

### 失敗パターン① あいまいな指示が「予測」を誘発する

「バグを探して」という指示に対してClaudeは**それらしい指摘を生成する**ことに最適化してしまう。実際に動作するコードでも「潜在的な問題として〜」という形式で指摘を作り出す。false positiveが大量発生する原因がここにある。

**悪いプロンプト例:**
```
以下のPythonコードのバグを見つけてください。
```

**Claudeの出力（問題あり）:**
```
1. 変数名が不明確な可能性があります
2. エラーハンドリングが不十分かもしれません
3. パフォーマンス上の懸念があります
...（根拠のない指摘が続く）
```

### 失敗パターン② スコープが広すぎてノイズが増える

「コードレビューして」という指示は、スタイル・設計・パフォーマンス・セキュリティ・テストカバレッジ、あらゆる観点が混在した回答を生成する。優先度のない大量の指摘は、実際には**何も直されない**という結果を招く。

### 失敗パターン③ コンテキスト不足で的外れになる

チームの規約、アーキテクチャの制約、このPRが解決しようとしている問題を知らずにレビューするのは、コードベースを一切知らない外部エンジニアにいきなりPRを投げるのと同じだ。「なぜこう書いているのか」という文脈なしのレビューはほぼ役に立たない。

## 5.2 4要素でプロンプトを構造化する

効果的なプロンプトはRole・Task・Format・Constraintの4要素で構成される。

```python
# prompt_builder.py
from dataclasses import dataclass, field


@dataclass
class ReviewPromptConfig:
    """レビュープロンプトの設定."""

    # Role: Claudeに与える役割
    role: str = "あなたは10年以上の経験を持つシニアバックエンドエンジニアです。"

    # Task: 具体的なタスク（観点を明示）
    focus_areas: list[str] = field(default_factory=lambda: [
        "バグ・クラッシュリスク",
        "セキュリティ脆弱性",
        "パフォーマンス問題",
    ])

    # Format: 出力形式の指定
    output_format: str = "json"  # "json" | "markdown"

    # Constraint: 制約条件
    max_issues: int = 10
    min_severity: str = "medium"  # "low" | "medium" | "high" | "critical"
    language: str = "Python"

    # Context: チーム固有の情報
    team_conventions: list[str] = field(default_factory=list)
    architecture_notes: str = ""


def build_review_prompt(
    diff_text: str,
    config: ReviewPromptConfig,
) -> str:
    """設定からレビュープロンプトを生成する."""

    convention_section = ""
    if config.team_conventions:
        conventions = "\n".join(f"- {c}" for c in config.team_conventions)
        convention_section = f"""
## チーム規約（これに従ってコードを評価してください）
{conventions}
"""

    arch_section = ""
    if config.architecture_notes:
        arch_section = f"""
## アーキテクチャ情報
{config.architecture_notes}
"""

    focus_list = "\n".join(
        f"{i+1}. {area}" for i, area in enumerate(config.focus_areas)
    )

    severity_constraint = (
        f"{config.min_severity} 以上の深刻度のものだけを報告してください。"
    )
    max_constraint = f"指摘は最大{config.max_issues}件に絞ってください。"

    return f"""## あなたの役割
{config.role}
{convention_section}{arch_section}
## レビュー対象のdiff
```diff
{diff_text}
```

## レビュー観点（優先度順）
{focus_list}

## 制約条件
- {severity_constraint}
- {max_constraint}
- 推測ではなく、diffで確認できる問題のみ指摘してください
- 「〜かもしれない」「〜の可能性がある」という指摘は禁止です

## 出力形式
以下のJSON形式で回答してください。余分なテキストは不要です。

```json
{{
  "issues": [
    {{
      "severity": "critical|high|medium|low",
      "category": "bug|security|performance|design",
      "file": "ファイルパス",
      "line_hint": 行番号または null,
      "description": "問題の説明（具体的に）",
      "evidence": "diffのどの部分が問題か（コード引用）",
      "suggestion": "具体的な修正案"
    }}
  ],
  "summary": "変更の概要（2文以内）",
  "overall_quality": "good|acceptable|needs_work"
}}
```
"""
```

## 5.3 観点別プロンプトテンプレート

レビュー観点ごとに専用プロンプトを持つと、汎用プロンプトより精度が上がる。

### セキュリティレビュープロンプト

```python
SECURITY_REVIEW_PROMPT = """
あなたはセキュリティ専門のコードレビュアーです。OWASPのTop 10と
CWE（Common Weakness Enumeration）に精通しています。

以下のdiffに含まれる**セキュリティ上の問題のみ**を報告してください。
設計や可読性の問題は範囲外です。

確認するべき脆弱性カテゴリ:
- A01: アクセス制御の不備（認証・認可のバイパス）
- A02: 暗号化の失敗（弱い暗号、ハードコードされた秘密情報）
- A03: インジェクション（SQL、OS、LDAP、XSS）
- A07: 認証・セッション管理の失敗
- A09: セキュリティログ・監視の失敗

## diff
```diff
{diff_text}
```

## 出力ルール
1. 確実に問題であるものだけ報告する（diffで証拠が確認できること）
2. 各指摘にCWE IDを含めること
3. 修正案は動作するコードで示すこと
4. 問題がない場合は `{{"issues": [], "verdict": "no_security_issues"}}` を返す

```json
{{
  "issues": [
    {{
      "cwe_id": "CWE-89",
      "severity": "critical|high|medium|low",
      "file": "ファイルパス",
      "vulnerable_code": "問題のあるコード断片",
      "attack_scenario": "攻撃シナリオの説明",
      "fix": "修正後のコード"
    }}
  ],
  "verdict": "has_issues|no_security_issues"
}}
```
"""
```

### パフォーマンスレビュープロンプト

```python
PERFORMANCE_REVIEW_PROMPT = """
あなたはバックエンドパフォーマンスの専門家です。
以下のdiffに含まれる**パフォーマンス上の問題のみ**を報告してください。

特に注目する問題:
- N+1クエリ（ループ内でのDBアクセス）
- インデックスが効かないクエリパターン
- メモリリーク（大きなオブジェクトの保持）
- 不要な繰り返し計算（ループ内での同一計算）
- 同期処理にすべき非同期処理（またはその逆）

## diff
```diff
{diff_text}
```

各指摘には必ず「計測方法」を含めてください。
推測でパフォーマンス問題を報告することは禁止です。

```json
{{
  "issues": [
    {{
      "type": "n_plus_1|memory_leak|redundant_computation|blocking_io|other",
      "severity": "high|medium|low",
      "file": "ファイルパス",
      "description": "問題の説明",
      "worst_case": "最悪ケースの影響（例: Nが1000件の場合1000クエリが発生）",
      "fix": "修正案",
      "how_to_measure": "計測方法（例: cProfile、EXPLAIN ANALYZE）"
    }}
  ]
}}
```
"""
```

## 5.4 A/Bテストで判明した「効くプロンプト vs 効かないプロンプト」

実際のPR 200件でA/Bテストを実施した結果を示す。評価基準は「開発者が有益と感じた指摘の割合（True Positive率）」と「不要な指摘の割合（False Positive率）」だ。

| 要素 | 効かないプロンプト | 効くプロンプト | TP率変化 |
|-----|----------------|--------------|---------|
| 指示の粒度 | 「レビューして」 | 「N+1クエリとSQL Injectionのみ確認して」 | +34% |
| 証拠の要求 | なし | 「diffで確認できる問題のみ」 | FP率 -41% |
| 修正案の形式 | 「〜を検討してください」 | 「動作するコードで修正案を示して」 | 有用性 +28% |
| 可能性表現の禁止 | なし | 「〜かもしれない は禁止」 | FP率 -23% |

最も効果的だったのは**「diffで確認できる問題のみ報告する」**という制約だ。これ1行追加するだけでFalse Positive率が大幅に低下した。Claudeは「問題がありそうなパターン」を記憶しており、コードを見るとそれを当てはめようとする傾向がある。この制約によって証拠なしの指摘を抑制できる。

## 5.5 false positiveを減らすプロンプト技術

```python
# false_positive_reducer.py

# ❌ 悪い例：あいまいな指示
BAD_PROMPT = """
このコードに問題がないか確認してください。
潜在的なバグや改善点を教えてください。
"""

# ✅ 良い例：制約を明示した指示
GOOD_PROMPT = """
以下の制約を**厳守**してください:

1. diffに明示的に存在するコードの問題のみ報告する
   - 削除されたコード（- 行）は報告対象外
   - 変更されていないコンテキスト行は報告対象外

2. 以下のカテゴリは報告しない:
   - コーディングスタイルの好み（変数名の長さ等）
   - 「〜のほうがよいかもしれない」レベルの提案
   - フレームワークのドキュメントに書かれているbest practice一般論

3. 報告する問題には必ず:
   - 該当するコードを引用すること
   - なぜ問題なのかを技術的に説明すること
   - 具体的な修正コードを示すこと

問題が見つからない場合は正直に「問題なし」と報告してください。
"""
```

## 5.6 コンテキスト注入——チーム規約をプロンプトに含める

チームのコーディング規約やアーキテクチャ情報をプロンプトに注入することで、レビュー精度が劇的に向上する。

```python
# context_injector.py
import os
from pathlib import Path


def load_team_context(repo_root: str) -> dict[str, str]:
    """リポジトリからチームのコンテキストを収集する."""
    root = Path(repo_root)
    context = {}

    # コーディング規約
    for filename in [".github/CONTRIBUTING.md", "CONVENTIONS.md", "docs/coding-style.md"]:
        filepath = root / filename
        if filepath.exists():
            context["conventions"] = filepath.read_text(encoding="utf-8")[:3000]
            break

    # アーキテクチャドキュメント
    for filename in ["docs/architecture.md", "ARCHITECTURE.md", "docs/design.md"]:
        filepath = root / filename
        if filepath.exists():
            context["architecture"] = filepath.read_text(encoding="utf-8")[:3000]
            break

    # 使用しているフレームワーク（requirements.txt / pyproject.toml から推定）
    req_file = root / "requirements.txt"
    if req_file.exists():
        context["dependencies"] = req_file.read_text(encoding="utf-8")[:1000]

    return context


def build_context_section(context: dict[str, str]) -> str:
    """コンテキスト情報をプロンプトセクションに変換する."""
    sections = []

    if "conventions" in context:
        sections.append(
            f"## チームのコーディング規約（抜粋）\n{context['conventions']}"
        )

    if "architecture" in context:
        sections.append(
            f"## アーキテクチャ概要\n{context['architecture']}"
        )

    if "dependencies" in context:
        sections.append(
            f"## 使用ライブラリ\n```\n{context['dependencies']}\n```"
        )

    if not sections:
        return ""

    return "\n\n".join(sections) + "\n\n"


# 使用例
def create_context_aware_prompt(diff_text: str, repo_root: str) -> str:
    context = load_team_context(repo_root)
    context_section = build_context_section(context)

    return f"""あなたは以下のプロジェクトのコードレビュアーです。

{context_section}## レビュー対象のdiff
```diff
{diff_text}
```

上記のチーム規約・アーキテクチャに照らして問題点を指摘してください。
一般的なbest practiceではなく、このプロジェクトの文脈で問題になる点を優先してください。
"""
```

## 🚨 本番落とし穴

### 落とし穴① プロンプトが長くなるほどコストが上がる

コンテキスト注入は有効だが、規約ドキュメントを全文入れると入力トークンが増えてコストが跳ね上がる。対策として**関連セクションだけを抽出**する方法が有効だ。変更されたファイルの言語・フレームワークに関連するセクションだけを選択的に注入する。

### 落とし穴② プロンプトのバージョン管理を怠る

プロンプトはコードと同様にバージョン管理が必要だ。プロンプトを変更するたびにレビュー品質が変わるため、変更履歴を追えない状態になると問題発生時に原因を特定できなくなる。プロンプトテンプレートをGitで管理し、バージョン番号をコメントに付けることを強く推奨する。

## 実務Tips

**段階的にプロンプトを改善する:** 最初から完璧なプロンプトを目指すより、まず動かして指摘の品質を毎週レビューするほうが現実的だ。「今週の false positive ベスト3」をチームで共有し、プロンプトに制約を追加していくサイクルが効果的だった。

**プロンプトのユニットテスト:** 既知のバグを含むテストdiffを作成し、プロンプト変更のたびに検出できるかを確認する「プロンプトユニットテスト」を導入すると退行を防げる。

## まとめ

- 「バグを探して」は根拠なき指摘を生む。観点・制約・フォーマットを明示する
- Role・Task・Format・Constraint の4要素でプロンプトを構造化する
- 「diffで確認できる問題のみ」という制約が false positive を最も効果的に削減する
- チームの規約・アーキテクチャをコンテキストとして注入することでレビューが文脈を理解する
- プロンプトもコードと同様にバージョン管理する
