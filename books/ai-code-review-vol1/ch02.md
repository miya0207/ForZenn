## 2.0 AIはどうやってコードを「読む」のか

AIコードレビューには2つの仕組みが組み合わさっている。

**①パターン検出（Pattern Matching）**
数百万のコードサンプルから学習した「危険なパターン」をコード内に見つける。
`f"SELECT...{変数}"` → SQLi、`def f(x=[])` → ミュータブルデフォルト引数、など。

**②LLM推論（Reasoning）**
コードの意図を推測し、「このパターンがなぜ危険か」「どう修正すべきか」を言語的に推論する。
単純なルールマッチではなく、文脈を踏まえた説明ができる点がLintツールとの違い。

```python
# LintツールとAIの違いの例
# flake8: "E501 line too long (120 > 79 characters)" → ルールベース
# Claude: "このループはN+1クエリを発生させています。SQLのJOINに書き換えてください" → 意味理解
```

**AIの限界（重要）**
- **幻覚（Hallucination）**: 自信を持って間違ったことを言う
- **知識の古さ**: 学習データのカットオフ以降の脆弱性は知らない
- **文脈の欠如**: チームの規約・ビジネスルールは知らない
- **非決定性**: 同じコードに毎回同じ指摘が返るとは限らない

これらの限界は第4章・第11章で詳しく扱う。

## 2.1 AIが検出できるカテゴリ①：バグパターン（論理エラー・型エラー）

バグの多くは「コードの意図と実装のズレ」から生まれる。AIは実装の構造を解析し、一般的なバグパターンに該当するコードを検出できる。

### よく見つかるバグパターン10選

```python
# ── Bug 1: Mutable デフォルト引数 ─────────────────────────────
def add_item(item: str, items: list = []) -> list:
    items.append(item)
    return items

# 問題: items は全ての呼び出しで同じリストを共有する
add_item("a")  # ["a"]
add_item("b")  # ["a", "b"] ← "a" が残っている！

# 修正
def add_item(item: str, items: list | None = None) -> list:
    if items is None:
        items = []
    items.append(item)
    return items


# ── Bug 2: 整数除算の意図しない切り捨て ─────────────────────────
def average(scores: list[int]) -> float:
    return sum(scores) / len(scores)  # OK (Python 3)

# Python 2 的な書き方が混入した場合
def average_buggy(scores):
    total = 0
    for s in scores:
        total += s
    return total / len(scores)  # Python 2 なら整数除算 → 0になる可能性


# ── Bug 3: 例外の種類を取り違える ──────────────────────────────
try:
    data = json.loads(response.text)
    value = data["key"]
except Exception:  # 広すぎる! どんな例外も握り潰す
    return None

# 修正: 例外を具体的に
try:
    data = json.loads(response.text)
    value = data["key"]
except json.JSONDecodeError:
    logger.error("JSONパース失敗")
    return None
except KeyError:
    logger.error("キー 'key' が存在しない")
    return None


# ── Bug 4: 浮動小数点の等値比較 ────────────────────────────────
price = 0.1 + 0.2
if price == 0.3:  # False になる！ (0.30000000000000004)
    apply_discount()

# 修正
import math
if math.isclose(price, 0.3, rel_tol=1e-9):
    apply_discount()


# ── Bug 5: 文字列フォーマットのオフバイワン ────────────────────
items = ["a", "b", "c"]
result = ""
for i in range(len(items)):
    result += items[i]
    if i < len(items):  # バグ: i は常に len(items) 未満
        result += ", "  # 最後の "," が消えない
# result = "a, b, c, "

# 修正
result = ", ".join(items)  # "a, b, c"
```

## 2.2 AIが検出できるカテゴリ②：セキュリティ脆弱性

セキュリティの問題は「パターンが明確」なため、AIによる自動検出の精度が特に高い。

### SQLインジェクション

```python
# ❌ 危険
def get_user(username: str):
    query = f"SELECT * FROM users WHERE username = '{username}'"
    return db.execute(query)

# 攻撃例: username = "admin' OR '1'='1"
# → SELECT * FROM users WHERE username = 'admin' OR '1'='1'
# → 全ユーザーが返る

# ✅ 安全: パラメータ化クエリ
def get_user(username: str):
    return db.execute(
        "SELECT * FROM users WHERE username = ?",
        (username,)
    )

# ✅ 安全: ORM（SQLAlchemy）
def get_user(username: str):
    return session.query(User).filter(User.username == username).first()
```

### コマンドインジェクション

```python
# ❌ 危険
import subprocess

def convert_image(filename: str):
    os.system(f"convert {filename} output.png")  # シェルインジェクション
    # filename = "input.jpg; rm -rf /"  → 全消去

# ✅ 安全: リストで渡す
def convert_image(filename: str):
    subprocess.run(
        ["convert", filename, "output.png"],  # シェルを経由しない
        check=True,
        capture_output=True,
    )
```

### パストラバーサル

```python
# ❌ 危険
from pathlib import Path

def read_file(filename: str) -> str:
    base_dir = Path("/var/www/uploads")
    return (base_dir / filename).read_text()
    # filename = "../../etc/passwd" → システムファイルが読まれる

# ✅ 安全: パスの正規化と検証
def read_file(filename: str) -> str:
    base_dir = Path("/var/www/uploads").resolve()
    target = (base_dir / filename).resolve()

    # base_dir の外に出ていないか確認
    if not str(target).startswith(str(base_dir)):
        raise ValueError(f"不正なパス: {filename}")

    return target.read_text()
```

## 2.3 AIが検出できるカテゴリ③：パフォーマンス問題

### N+1クエリ（最もよく見つかる問題）

```python
# ❌ N+1クエリ問題
def get_dashboard_data(user_ids: list[int]) -> list[dict]:
    result = []
    users = db.query(User).filter(User.id.in_(user_ids)).all()

    for user in users:
        # ← ループごとにSQLが実行される（N+1）
        posts = db.query(Post).filter(Post.user_id == user.id).all()
        result.append({
            "user": user.name,
            "post_count": len(posts),
        })
    return result

# ✅ 修正: JOIN または サブクエリ
from sqlalchemy import func

def get_dashboard_data(user_ids: list[int]) -> list[dict]:
    # 1回のSQLで全データを取得
    rows = (
        db.query(User, func.count(Post.id).label("post_count"))
        .outerjoin(Post, Post.user_id == User.id)
        .filter(User.id.in_(user_ids))
        .group_by(User.id)
        .all()
    )
    return [{"user": user.name, "post_count": count} for user, count in rows]
```

### 不必要なリスト生成

```python
# ❌ 全件をリストに展開してからカウント（メモリ非効率）
def count_active_users(users):
    active = [u for u in users if u.is_active]  # 巨大リスト作成
    return len(active)

# ✅ ジェネレータ式 + sum
def count_active_users(users):
    return sum(1 for u in users if u.is_active)  # メモリO(1)

# ✅ または filter
def count_active_users(users):
    return sum(1 for _ in filter(lambda u: u.is_active, users))
```

### ループ内の重複計算

```python
# ❌ ループ内で毎回計算
import re

def process_logs(log_lines: list[str]) -> list[str]:
    pattern = re.compile(r"\d{4}-\d{2}-\d{2}")  # OK（ループ外）
    results = []
    for line in log_lines:
        config = load_config()  # ← ループ内！毎回ファイル読み込み
        threshold = config["error_threshold"]
        if pattern.search(line):
            results.append(line)
    return results

# ✅ ループ外に移動
def process_logs(log_lines: list[str]) -> list[str]:
    pattern = re.compile(r"\d{4}-\d{2}-\d{2}")
    config = load_config()   # ← ループ外
    threshold = config["error_threshold"]
    return [line for line in log_lines if pattern.search(line)]
```

## 2.4 AIが検出できるカテゴリ④：スタイルと可読性

スタイルの問題は「動作に影響しないが、保守性を下げる」問題だ。Lintツールが取り逃がすパターンをAIは指摘できる。

```python
# ❌ 否定条件の重ね使い（読みにくい）
def process_item(item):
    if not item is None:      # → if item is not None:
        if not len(item) == 0:  # → if item:
            if not item.is_deleted != True:  # → if not item.is_deleted:
                return item.process()

# ✅ 整理後
def process_item(item):
    if item and not item.is_deleted:
        return item.process()


# ❌ マジックナンバー
def calculate_fee(amount: float) -> float:
    if amount > 10000:
        return amount * 0.05
    elif amount > 5000:
        return amount * 0.03
    return amount * 0.01

# ✅ 意味のある名前
FEE_TIER_HIGH_THRESHOLD = 10_000
FEE_TIER_MID_THRESHOLD = 5_000
FEE_RATE_HIGH = 0.05
FEE_RATE_MID = 0.03
FEE_RATE_BASE = 0.01

def calculate_fee(amount: float) -> float:
    if amount > FEE_TIER_HIGH_THRESHOLD:
        return amount * FEE_RATE_HIGH
    elif amount > FEE_TIER_MID_THRESHOLD:
        return amount * FEE_RATE_MID
    return amount * FEE_RATE_BASE


# ❌ 長すぎる関数（単一責任原則違反）
def process_order(order_id: int) -> dict:
    # 100行の処理...
    # → 検証・計算・保存・通知 が1関数に混在

# ✅ 分割
def process_order(order_id: int) -> dict:
    order = fetch_order(order_id)
    validate_order(order)
    total = calculate_total(order)
    save_order(order, total)
    notify_user(order)
    return {"order_id": order_id, "total": total}
```

## 2.5 AIが検出できるカテゴリ⑤：重複コード（DRY原則違反）

重複コードはAIが発見しやすい問題の一つだ。コードの意味より構造の類似性を見るのが得意なAIにとって、コピペは明確に検出できる。

```python
# ❌ 重複コード（明らかなコピペ）
def send_welcome_email(user_id: int) -> bool:
    user = db.query(User).filter(User.id == user_id).first()
    if not user:
        logger.error(f"User {user_id} not found")
        return False

    try:
        smtp.send(
            to=user.email,
            subject="ようこそ！",
            body=f"こんにちは、{user.name}さん。"
        )
        logger.info(f"Welcome email sent to {user.email}")
        return True
    except SMTPException as e:
        logger.error(f"Email送信失敗: {e}")
        return False

def send_reset_email(user_id: int, token: str) -> bool:
    user = db.query(User).filter(User.id == user_id).first()
    if not user:
        logger.error(f"User {user_id} not found")  # ← コピペ

        smtp.send(
            to=user.email,
            subject="パスワードリセット",
            body=f"リセットURL: /reset?token={token}"

        logger.info(f"Reset email sent to {user.email}")  # ← コピペ
        return True
```

## 2.6 5カテゴリを一括チェックする実装（コピペで動く）

```python
"""
ai_reviewer/checkers.py — 5カテゴリの一括チェック実装
"""
from dataclasses import dataclass
from enum import Enum
import anthropic

class Severity(Enum):
    CRITICAL = "🔴"
    WARNING  = "🟡"
    INFO     = "🔵"

@dataclass
class ReviewComment:
    category: str
    severity: Severity
    line_hint: str
    message: str
    suggestion: str = ""

MASTER_REVIEW_PROMPT = """
あなたは経験豊富なPythonシニアエンジニアです。
以下の5カテゴリでコードをレビューし、問題を列挙してください。

【カテゴリと重要度】
1. バグ (🔴 重大 / 🟡 警告)
   - ミュータブルデフォルト引数、浮動小数点比較、例外握り潰し
2. セキュリティ (🔴 重大)
   - SQLインジェクション、コマンドインジェクション、パストラバーサル
3. パフォーマンス (🟡 警告)
   - N+1クエリ、ループ内重複計算、不必要なリスト生成
4. 可読性 (🔵 情報)
   - マジックナンバー、否定条件の重ね使い、長すぎる関数
5. 重複 (🟡 警告)
   - 明らかなコピペ、共通化できるロジック

【出力形式 (JSON)】
{
  "summary": "総評（1〜2文）",
  "comments": [
    {
      "category": "バグ|セキュリティ|パフォーマンス|可読性|重複",
      "severity": "🔴|🟡|🔵",
      "location": "関数名または行の説明",
      "problem": "問題の説明",
      "fix": "修正例（コードを含めてよい）"
    }
  ],
  "overall_score": 1-10  // 10が最高品質
}

問題がない場合はcommentsを空配列にしてください。
"""

def review_code(code: str, model: str = "claude-haiku-4-5") -> dict:
    """5カテゴリ一括レビューを実行する"""
    client = anthropic.Anthropic()

    response = client.messages.create(
        model=model,
        max_tokens=4096,
        messages=[{
            "role": "user",
            "content": f"{MASTER_REVIEW_PROMPT}\n\n```python\n{code}\n```"
        }]
    )

    import json, re
    text = response.content[0].text
    # JSONブロックを抽出
    match = re.search(r'\{.*\}', text, re.DOTALL)
    if match:
        return json.loads(match.group())
    return {"summary": text, "comments": [], "overall_score": 0}
```

## 実務Tips

**Tip 1: カテゴリごとにモデルを使い分ける**
バグ・スタイル → Haiku（速度優先）
セキュリティ・アーキテクチャ → Sonnet（精度優先）

**Tip 2: ファイルサイズでチャンクを分割する**
1,000行を超えるファイルは関数単位に分割してレビューする。一度に全部送ると精度が下がる。

**Tip 3: false positive を記録してプロンプトを改善する**
「AIが指摘したが問題ではなかった」事例を記録し、プロンプトに「以下は除外してください」として追記する。

## まとめ

AIが高精度で検出できる5カテゴリを整理した：

1. **バグ** — ミュータブルデフォルト引数、浮動小数点比較など、構造から判定できる問題
2. **セキュリティ** — OWASP Top 10、特にインジェクション系は検出率が高い
3. **パフォーマンス** — N+1クエリ、ループ内重複計算は明確なパターンがある
4. **可読性** — マジックナンバー、長関数など、Lintツールを補完する指摘
5. **重複** — 構造的類似性から重複を検出するのはAIが得意

次章では、これらを「AIが検出すること」として、人間レビュアーが担うべき「AIでは検出しにくい視点」を対比して解説する。
