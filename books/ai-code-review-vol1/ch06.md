# 第6章: 設計レビュー — アーキテクチャの問題を早期発見

:::message
**この章で学べること**
- AIが設計問題を検出できる範囲と限界の正確な把握
- SOLID原則をプロンプトに変換してAIレビューを構造化する方法
- ASTで依存グラフを生成してClaudeに渡す実装と、循環依存の自動検出
:::

## 6.1 設計図なしにAIが設計問題を検出する限界

コードレビューにAIを活用しようとしたとき、最初につまずくのが「設計問題の検出」だ。

AIはdiffの中の**ローカルなパターン**は検出できる。例えばGod Classの兆候（メソッドが30個以上）や、メソッドが長すぎること、引数が多すぎることは1ファイル内で完結するのでdiffから判断できる。

しかし**システム全体の文脈が必要な設計問題**はdiffだけでは判断できない。

```
AIが検出できる設計問題（diffのみで判断可能）
  ✅ God Class（1クラスにメソッドが集中）
  ✅ Long Method（1メソッドが100行超）
  ✅ Feature Envy（他クラスのメソッドを多用）
  ✅ Magic Number（説明なしの数値リテラル）
  ✅ 重複コード（同一処理の繰り返し）

AIが検出できない設計問題（システム全体の知識が必要）
  ❌ レイヤーアーキテクチャの違反（どのレイヤーが何かを知らない）
  ❌ ドメインモデルの整合性（ドメイン知識がない）
  ❌ マイクロサービス間の責務の重複
  ❌ 将来の機能追加に対する拡張性の評価
```

この限界を正確に把握した上で、「AIが得意な範囲」に絞ってレビューを設計することが重要だ。本章では**依存関係情報をASTから生成してAIに渡す**ことで、AIの検出範囲を拡張するアプローチを取る。

## 6.2 SOLID原則ベースのAIレビュープロンプト

SOLID原則は抽象的な概念だが、コードの具体的なシグナルに変換できる。

```python
# solid_reviewer.py
from dataclasses import dataclass


@dataclass
class SolidViolationCheck:
    """SOLID各原則の違反シグナルの定義."""

    srp_signals: list[str]   # 単一責任原則
    ocp_signals: list[str]   # 開放閉鎖原則
    lsp_signals: list[str]   # リスコフの置換原則
    isp_signals: list[str]   # インターフェース分離原則
    dip_signals: list[str]   # 依存性逆転原則


DEFAULT_SOLID_CHECK = SolidViolationCheck(
    srp_signals=[
        "1つのクラスに5つ以上の異なる種類の処理が混在している",
        "クラス名に 'And', 'Manager', 'Handler', 'Util' が含まれる",
        "プライベートメソッドが10個以上存在する",
        "クラスのインポートにDB・HTTP・ファイルIOが混在している",
    ],
    ocp_signals=[
        "新しい種別を追加するためにif/elseやswitchを修正している",
        "機能追加のために既存クラスのメソッドシグネチャを変更している",
        "文字列リテラルでの分岐（`if type == 'email':`）",
    ],
    lsp_signals=[
        "サブクラスで基底クラスのメソッドが `raise NotImplementedError` になっている",
        "オーバーライドしたメソッドの事前条件が基底クラスより厳しくなっている",
        "基底クラスの戻り値型と異なる型をサブクラスが返している",
    ],
    isp_signals=[
        "インターフェース（Abstract Base Class）のメソッドの多くが実装クラスで `pass` または `raise NotImplementedError`",
        "使用しないメソッドを実装するために空の実装がある",
    ],
    dip_signals=[
        "高レベルモジュールが低レベルの具体クラスを直接インスタンス化している（`UserRepository()` を直接 `new`）",
        "依存する具体クラスをコンストラクタ内で生成している",
        "テストのためにモンキーパッチが必要になる構造",
    ],
)


def build_solid_review_prompt(diff_text: str, check: SolidViolationCheck) -> str:
    """SOLID原則レビュープロンプトを生成する."""

    def format_signals(signals: list[str]) -> str:
        return "\n".join(f"  - {s}" for s in signals)

    return f"""あなたはSOLID原則に精通したソフトウェアアーキテクトです。
以下のdiffをレビューし、SOLID原則の違反を検出してください。

## 各原則の違反シグナル

### S: 単一責任原則 (SRP)
{format_signals(check.srp_signals)}

### O: 開放閉鎖原則 (OCP)
{format_signals(check.ocp_signals)}

### L: リスコフの置換原則 (LSP)
{format_signals(check.lsp_signals)}

### I: インターフェース分離原則 (ISP)
{format_signals(check.isp_signals)}

### D: 依存性逆転原則 (DIP)
{format_signals(check.dip_signals)}

## diff
```diff
{diff_text}
```

## 制約
- diffに明確に現れているシグナルのみ報告する
- 「〜の可能性がある」という指摘は禁止。確証があるものだけ報告する
- 各指摘にはdiffの該当コードを引用すること

```json
{{
  "violations": [
    {{
      "principle": "SRP|OCP|LSP|ISP|DIP",
      "severity": "high|medium|low",
      "file": "ファイルパス",
      "evidence": "問題のあるコード断片",
      "explanation": "なぜ違反なのか（1〜2文）",
      "refactoring": "リファクタリング案"
    }}
  ],
  "overall_design_quality": "good|acceptable|needs_refactoring"
}}
```
"""
```

## 6.3 ASTで依存グラフを生成してClaudeに渡す

Pythonの `ast` モジュールを使ってファイル間の依存関係グラフを生成し、Claudeに渡すことで設計レビューの精度を上げる。

```python
# dependency_analyzer.py
import ast
import os
from dataclasses import dataclass, field
from pathlib import Path


@dataclass
class ModuleDependency:
    """モジュールの依存情報."""
    module: str
    imports: list[str] = field(default_factory=list)
    from_imports: list[str] = field(default_factory=list)


@dataclass
class DependencyGraph:
    """プロジェクト全体の依存グラフ."""
    nodes: list[str]          # モジュール名一覧
    edges: list[tuple[str, str]]  # (from, to) のエッジ
    circular_deps: list[list[str]]  # 循環依存のサイクル
    metrics: dict             # 各モジュールの統計情報


class DependencyAnalyzer:
    """ASTによる依存関係解析."""

    def __init__(self, project_root: str):
        self.project_root = Path(project_root)
        self.package_name = self._detect_package_name()

    def _detect_package_name(self) -> str:
        """トップレベルのPythonパッケージ名を検出する."""
        for item in self.project_root.iterdir():
            if item.is_dir() and (item / "__init__.py").exists():
                return item.name
        return ""

    def _extract_imports(self, filepath: Path) -> ModuleDependency:
        """ファイルからimport文を抽出する."""
        try:
            source = filepath.read_text(encoding="utf-8")
            tree = ast.parse(source)
        except (SyntaxError, UnicodeDecodeError):
            return ModuleDependency(module=str(filepath))

        module_name = self._path_to_module(filepath)
        dep = ModuleDependency(module=module_name)

        for node in ast.walk(tree):
            if isinstance(node, ast.Import):
                for alias in node.names:
                    dep.imports.append(alias.name)
            elif isinstance(node, ast.ImportFrom):
                if node.module:
                    dep.from_imports.append(node.module)

        return dep

    def _path_to_module(self, filepath: Path) -> str:
        """ファイルパスをモジュール名に変換する."""
        rel = filepath.relative_to(self.project_root)
        parts = list(rel.parts)
        if parts[-1] == "__init__.py":
            parts = parts[:-1]
        elif parts[-1].endswith(".py"):
            parts[-1] = parts[-1][:-3]
        return ".".join(parts)

    def _is_internal(self, module_name: str) -> bool:
        """プロジェクト内部のモジュールか判定する."""
        if not self.package_name:
            return False
        return module_name.startswith(self.package_name)

    def analyze(self) -> DependencyGraph:
        """プロジェクト全体の依存グラフを構築する."""
        all_deps: dict[str, list[str]] = {}

        for py_file in self.project_root.rglob("*.py"):
            dep = self._extract_imports(py_file)
            internal_imports = [
                m for m in dep.imports + dep.from_imports
                if self._is_internal(m)
            ]
            all_deps[dep.module] = internal_imports

        nodes = list(all_deps.keys())
        edges = [
            (src, dst)
            for src, dsts in all_deps.items()
            for dst in dsts
            if dst in all_deps
        ]

        circular_deps = self._find_cycles(all_deps)

        metrics = {
            module: {
                "fan_out": len(deps),  # このモジュールが依存する数
                "fan_in": sum(
                    1 for dsts in all_deps.values() if module in dsts
                ),  # このモジュールに依存する数
            }
            for module, deps in all_deps.items()
        }

        return DependencyGraph(
            nodes=nodes,
            edges=edges,
            circular_deps=circular_deps,
            metrics=metrics,
        )

    def _find_cycles(self, graph: dict[str, list[str]]) -> list[list[str]]:
        """DFSで循環依存を検出する."""
        visited = set()
        rec_stack = set()
        cycles = []

        def dfs(node: str, path: list[str]) -> None:
            visited.add(node)
            rec_stack.add(node)
            path.append(node)

            for neighbor in graph.get(node, []):
                if neighbor not in visited:
                    dfs(neighbor, path)
                elif neighbor in rec_stack:
                    # サイクルを発見
                    cycle_start = path.index(neighbor)
                    cycles.append(path[cycle_start:])

            path.pop()
            rec_stack.discard(node)

        for node in graph:
            if node not in visited:
                dfs(node, [])

        return cycles


def graph_to_prompt_section(graph: DependencyGraph) -> str:
    """依存グラフをプロンプトのセクションに変換する."""
    lines = ["## プロジェクトの依存グラフ情報\n"]

    # 高fan-outモジュール（依存が多い）
    high_fan_out = [
        (m, info["fan_out"])
        for m, info in graph.metrics.items()
        if info["fan_out"] >= 5
    ]
    if high_fan_out:
        high_fan_out.sort(key=lambda x: -x[1])
        lines.append("### 依存が多いモジュール (fan-out >= 5)")
        for module, count in high_fan_out[:10]:
            lines.append(f"- `{module}`: {count}モジュールに依存")
        lines.append("")

    # 高fan-inモジュール（多くから依存される）
    high_fan_in = [
        (m, info["fan_in"])
        for m, info in graph.metrics.items()
        if info["fan_in"] >= 5
    ]
    if high_fan_in:
        high_fan_in.sort(key=lambda x: -x[1])
        lines.append("### 多くのモジュールから依存されるモジュール (fan-in >= 5)")
        for module, count in high_fan_in[:10]:
            lines.append(f"- `{module}`: {count}モジュールから依存される")
        lines.append("")

    # 循環依存
    if graph.circular_deps:
        lines.append(f"### 循環依存 ({len(graph.circular_deps)}件)")
        for cycle in graph.circular_deps:
            lines.append(f"- {' → '.join(cycle)} → {cycle[0]}")
        lines.append("")
    else:
        lines.append("### 循環依存: なし\n")

    return "\n".join(lines)
```

## 6.4 循環依存の自動検出と `import-linter` との比較

循環依存の検出には `import-linter` というツールも使えるが、それぞれに特性がある。

| 観点 | ASTベース自装実装 | import-linter |
|-----|----------------|---------------|
| セットアップ | コードを書くだけ | 設定ファイルが必要 |
| 検出精度 | 静的解析（動的importは検出不可） | 同様 |
| CI統合 | 柔軟（AI連携可能） | `.importlinter` 設定で制御 |
| ルールのカスタマイズ | 完全自由 | 設定ベース |
| レポート形式 | 自由（AIへの入力に変換可能） | 固定形式 |

実務では両方を使い分ける方法が最も効果的だ。`import-linter` をCIでゲートに使い、違反があったときにASTの解析結果をClaudeに渡して「この循環依存を解消するリファクタリング案を提示して」と聞く構成がよい。

```bash
# import-linter の設定例
# .importlinter
[importlinter]
root_package = myapp

[importlinter:contract:no-circular]
name = No circular dependencies
type = forbidden
source_modules =
    myapp
forbidden_modules =
    myapp
```

## 6.5 設計レビューチェックリストをプロンプトに変換する

```python
# design_checklist.py
DESIGN_REVIEW_CHECKLIST = [
    # God Class
    {
        "name": "God Class",
        "signal": "クラスのパブリックメソッドが15個以上、または行数が500行以上",
        "why": "1つのクラスが多くの責任を持ちすぎており、変更の影響範囲が広くなる",
        "fix": "責任ごとに複数のクラスに分割する",
    },
    # Feature Envy
    {
        "name": "Feature Envy",
        "signal": "あるメソッドが自クラスのフィールドより他クラスのメソッド・フィールドを多用している",
        "why": "そのメソッドは本来、使用している他クラスに属するべき",
        "fix": "メソッドを使用先のクラスに移動する",
    },
    # Shotgun Surgery
    {
        "name": "Shotgun Surgery",
        "signal": "1つの機能変更が複数ファイル（5ファイル以上）に分散している",
        "why": "変更コストが高く、変更漏れのリスクが大きい",
        "fix": "関連する処理を1箇所にまとめる",
    },
    # Data Clumps
    {
        "name": "Data Clumps",
        "signal": "同じ引数の組み合わせ（例: first_name, last_name, email）が複数の関数に繰り返し現れる",
        "why": "関連データはオブジェクトにまとめるべき",
        "fix": "引数グループをデータクラスやNamedTupleにまとめる",
    },
]


def build_antipattern_prompt(diff_text: str, dependency_section: str = "") -> str:
    """アンチパターン検出プロンプトを生成する."""
    checklist_text = "\n\n".join(
        f"### {item['name']}\n"
        f"- **シグナル**: {item['signal']}\n"
        f"- **問題**: {item['why']}\n"
        f"- **対処**: {item['fix']}"
        for item in DESIGN_REVIEW_CHECKLIST
    )

    return f"""あなたは設計パターンとリファクタリングの専門家です。
以下のdiffを、既知の設計アンチパターンの観点でレビューしてください。

{dependency_section}

## 確認するアンチパターン

{checklist_text}

## diff
```diff
{diff_text}
```

## 重要な制約
- **必ずdiffに存在するコードから証拠を示すこと**
- diffに現れていないコードの問題は推測で報告しないこと
- 問題がない場合は `"antipatterns_found": []` を返すこと

```json
{{
  "antipatterns_found": [
    {{
      "pattern_name": "God Class|Feature Envy|Shotgun Surgery|Data Clumps",
      "severity": "high|medium|low",
      "file": "ファイルパス",
      "evidence": "問題を示すコード引用",
      "refactoring_hint": "具体的なリファクタリング案"
    }}
  ],
  "design_score": "A|B|C|D"
}}
```
"""
```

## 🚨 本番落とし穴

### 落とし穴① AST解析の例外処理を怠る

`ast.parse()` は構文エラーがあるとクラッシュする。マイグレーションファイルや自動生成コードに構文エラーが含まれるケースがあるため、必ず `try-except` でラップし、解析失敗ファイルをスキップする実装が必要だ。

### 落とし穴② fan-outの多いモジュールを過剰に問題視する

`utils.py` や `constants.py` はプロジェクト全体から参照される性質上、fan-inが自然に高くなる。これを「God Class」と誤認しないよう、モジュール名でフィルタリングするロジックをASTアナライザーに追加しておく。

## 実務Tips

**設計レビューは全PRに適用しない:** 設計問題の検出は処理時間もコストも高い。変更行数が50行以上のPR、または設計ファイル（`service.py`, `repository.py`, `model.py`）が変更されたPRに限定して実行すると費用対効果が高い。

## まとめ

- AIが設計問題を検出できる範囲はローカルなパターンに限られる。システム全体の文脈は別途提供が必要
- SOLID原則は「違反シグナル」に変換することでAIが検出しやすくなる
- ASTで依存グラフを生成してClaudeに渡すことで、ファイルをまたぐ問題を検出できる
- 循環依存は `import-linter` でゲートに使い、AIでリファクタリング案を生成する組み合わせが効果的
