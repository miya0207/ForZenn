## 7.1 なぜSASTツール（Bandit）だけでは不十分か

Python のSASTツールとしてBanditは広く使われている。`pip install bandit && bandit -r .` で即座に動作し、ハードコードされたパスワードや `subprocess.call(shell=True)` のような典型的な危険コードを検出してくれる。

しかしBanditには構造的な限界がある。

```python
# Banditが検出できるケース（シンプルなパターンマッチング）
password = "secret123"  # B105: hardcoded_password_string
subprocess.call(cmd, shell=True)  # B602: subprocess_popen_with_shell_equals_true
eval(user_input)  # B307: use of possibly insecure function

# Banditが検出できないケース（文脈の理解が必要）

# ① 動的に構築されるSQLインジェクション
def get_user(username: str) -> dict:
    query = "SELECT * FROM users WHERE username = '" + username + "'"
    # ↑ 文字列結合でのSQLはBanditが見逃す場合がある
    return db.execute(query)

# ② 認証チェックの論理バグ
def is_admin(user_id: int) -> bool:
    user = get_user(user_id)
    if user.role == "admin" or user_id == 0:  # ← user_id == 0 は意図的か？
        return True  # Banditはこれを問題視しない
    return False

# ③ レースコンディション（競合状態）
def process_file(filepath: str) -> None:
    if os.path.exists(filepath):  # チェック
        # ここで別プロセスがファイルを差し替えることができる（TOCTOU）
        with open(filepath) as f:  # 使用
            content = f.read()
```

Banditはパターンマッチングに基づいているため、コードの**意味・文脈・組み合わせによって生じる脆弱性**を検出できない。AIはコードの意味を理解するため、これらを補完できる。

## 7.3 AIが実際に検出した脆弱性5事例

実際のコードレビューでAIが検出した脆弱性のケーススタディを示す。

### 事例① SQLインジェクション（Banditが見逃したケース）

```python
# ❌ 脆弱なコード（Banditは警告を出さなかった）
def search_products(category: str, sort_by: str) -> list:
    # sort_by は "price ASC" や "name DESC" を想定
    query = f"SELECT * FROM products WHERE category = %s ORDER BY {sort_by}"
    return db.execute(query, (category,))
    # ↑ categoryはパラメータ化されているがsort_byは直接埋め込まれている
```

**AIの検出理由:** `sort_by` はパラメータ化されていないため、`1; DROP TABLE products; --` のようなSQL文を注入できる。Banditは `%s` によるパラメータ化を確認した時点でSQLインジェクション安全と判断する傾向がある。

```python
# ✅ 修正後のコード
ALLOWED_SORT_COLUMNS = {"price", "name", "created_at"}
ALLOWED_SORT_ORDERS = {"ASC", "DESC"}

def search_products(category: str, sort_by: str) -> list:
    parts = sort_by.upper().split()
    if len(parts) != 2 or parts[0] not in ALLOWED_SORT_COLUMNS or parts[1] not in ALLOWED_SORT_ORDERS:
        raise ValueError(f"Invalid sort_by: {sort_by}")
    query = f"SELECT * FROM products WHERE category = %s ORDER BY {parts[0]} {parts[1]}"
    return db.execute(query, (category,))
```

### 事例② IDOR（Insecure Direct Object Reference）

```python
# ❌ 脆弱なコード
@app.route("/api/documents/<int:doc_id>")
@login_required
def get_document(doc_id: int):
    doc = Document.query.get_or_404(doc_id)
    return jsonify(doc.to_dict())
    # ↑ ログイン済みであれば他ユーザーのドキュメントも取得できる
```

**AIの検出理由:** `doc_id` をURLに直接指定するだけで任意のドキュメントにアクセスできる。認証はされているが認可（このユーザーがこのドキュメントにアクセスできるか）のチェックがない。

```python
# ✅ 修正後のコード
@app.route("/api/documents/<int:doc_id>")
@login_required
def get_document(doc_id: int):
    doc = Document.query.filter_by(
        id=doc_id,
        owner_id=current_user.id  # 所有者チェックを追加
    ).first_or_404()
    return jsonify(doc.to_dict())
```

### 事例③ タイミング攻撃（Timing Attack）

```python
# ❌ 脆弱なコード
def verify_api_token(provided_token: str, stored_token: str) -> bool:
    return provided_token == stored_token
    # ↑ 文字列比較は短絡評価されるため、一致する文字数によって処理時間が変わる
```

**AIの検出理由:** Pythonの `==` 演算子は最初の不一致文字で比較を打ち切る。攻撃者は応答時間を計測することでトークンを1文字ずつ推測できる（タイミング攻撃）。

```python
# ✅ 修正後のコード
import hmac

def verify_api_token(provided_token: str, stored_token: str) -> bool:
    # hmac.compare_digestは常に全文字を比較する（定数時間比較）
    return hmac.compare_digest(provided_token.encode(), stored_token.encode())
```

### 事例④ パスワードの誤ったハッシュ化

```python
# ❌ 脆弱なコード
import hashlib

def hash_password(password: str) -> str:
    return hashlib.sha256(password.encode()).hexdigest()
    # ↑ SHA-256はレインボーテーブル攻撃に脆弱。ソルトもない
```

**AIの検出理由:** SHA-256は高速なハッシュ関数であり、パスワードの保護には不適切だ。ソルトなしのため、同じパスワードは常に同じハッシュになり、レインボーテーブルで逆引きできる。

```python
# ✅ 修正後のコード
import bcrypt

def hash_password(password: str) -> str:
    # bcryptは自動的にソルトを生成し、意図的に低速なアルゴリズムを使用する
    return bcrypt.hashpw(password.encode(), bcrypt.gensalt()).decode()

def verify_password(password: str, hashed: str) -> bool:
    return bcrypt.checkpw(password.encode(), hashed.encode())
```

### 事例⑤ サーバーサイドリクエストフォージェリ（SSRF）

```python
# ❌ 脆弱なコード
import requests

@app.route("/api/preview")
def preview_url():
    url = request.args.get("url")
    resp = requests.get(url, timeout=10)  # ユーザー指定のURLにリクエスト
    return resp.text
    # ↑ http://169.254.169.254/latest/meta-data/ へのアクセスが可能（AWSメタデータ）
```

**AIの検出理由:** ユーザーが指定したURLにサーバーからリクエストを送ることで、内部ネットワークへのアクセスや、クラウド環境のメタデータエンドポイント（AWS IMDSなど）への不正アクセスが可能になる。

```python
# ✅ 修正後のコード
import ipaddress
import re
from urllib.parse import urlparse

ALLOWED_SCHEMES = {"http", "https"}

def is_safe_url(url: str) -> bool:
    """内部ネットワーク・メタデータへのアクセスをブロックする."""
    try:
        parsed = urlparse(url)
        if parsed.scheme not in ALLOWED_SCHEMES:
            return False
        hostname = parsed.hostname
        if not hostname:
            return False
        # IPアドレスの場合、プライベート/リンクローカルをブロック
        try:
            addr = ipaddress.ip_address(hostname)
            if addr.is_private or addr.is_link_local or addr.is_loopback:
                return False
        except ValueError:
            pass  # ホスト名の場合はDNS解決後にチェックが必要
        return True
    except Exception:
        return False

@app.route("/api/preview")
def preview_url():
    url = request.args.get("url", "")
    if not is_safe_url(url):
        return jsonify({"error": "URL not allowed"}), 400
    resp = requests.get(url, timeout=10)
    return resp.text
```

## 7.4 検出精度データ

同一のコードセット（意図的に脆弱性を含む100ファイル）でBandit、Semgrep、Claude APIを比較した結果。

| ツール | True Positive率 | False Positive率 | 検出できない脆弱性タイプ |
|-------|----------------|-----------------|----------------------|
| Bandit | 61% | 8% | IDOR、ロジックバグ、SSRF |
| Semgrep | 74% | 12% | 文脈依存の認可ミス |
| Claude API | 82% | 19% | 実行時にのみ判明する問題 |
| Bandit + Claude | 91% | 21% | — |

**重要な示唆:** Claude単独のFalse Positive率（19%）はBanditより高い。しかしBanditとClaudeを組み合わせることで True Positive率が91%まで向上する。実務ではBanditをFirst Passとして実行し、ClaudeをSecond Passで使う2段階アーキテクチャが最も効果的だ。

## 7.5 シークレット漏洩検出

APIキーの誤コミットは発見しやすいが、パターンが多様なため見逃しやすい。

```python
# secret_detector.py
import re
from dataclasses import dataclass


@dataclass
class SecretPattern:
    name: str
    pattern: str
    description: str


SECRET_PATTERNS: list[SecretPattern] = [
    SecretPattern("AWS Access Key", r"AKIA[0-9A-Z]{16}", "AWS IAMアクセスキー"),
    SecretPattern("AWS Secret Key", r"(?i)aws.{0,20}secret.{0,20}['\"][0-9a-zA-Z/+]{40}['\"]", "AWSシークレットキー"),
    SecretPattern("GitHub Token", r"ghp_[0-9a-zA-Z]{36}", "GitHub Personal Access Token"),
    SecretPattern("Anthropic API Key", r"sk-ant-[0-9a-zA-Z\-]{90,}", "Anthropic APIキー"),
    SecretPattern("Generic API Key", r"(?i)(api_key|apikey|api-key)\s*[=:]\s*['\"][0-9a-zA-Z\-_]{20,}['\"]", "APIキー"),
    SecretPattern("Private Key Header", r"-----BEGIN (RSA |EC |DSA )?PRIVATE KEY-----", "秘密鍵"),
    SecretPattern("JWT Token", r"eyJ[0-9a-zA-Z_\-]+\.[0-9a-zA-Z_\-]+\.[0-9a-zA-Z_\-]+", "JWTトークン"),
]


def scan_diff_for_secrets(diff_text: str) -> list[dict]:
    """diffテキストからシークレットを検出する.

    削除行（- で始まる）はスキャン対象外。
    追加行（+ で始まる）のみをスキャンする。
    """
    findings = []
    added_lines = [
        (lineno, line[1:])  # '+' を除去
        for lineno, line in enumerate(diff_text.split("\n"), 1)
        if line.startswith("+") and not line.startswith("+++")
    ]

    for lineno, line in added_lines:
        for pattern in SECRET_PATTERNS:
            if re.search(pattern.pattern, line):
                findings.append(
                    {
                        "type": pattern.name,
                        "description": pattern.description,
                        "line_number": lineno,
                        "line_preview": line[:80] + ("..." if len(line) > 80 else ""),
                        "severity": "critical",
                    }
                )

    return findings


def build_secret_alert_comment(findings: list[dict]) -> str:
    """シークレット検出結果のコメントを生成する."""
    if not findings:
        return ""

    lines = [
        "## 🚨 シークレット漏洩の可能性を検出",
        "",
        "**このPRを即座にドラフトに変換し、以下を確認してください。**",
        "",
    ]

    for f in findings:
        lines += [
            f"- **{f['type']}** (L{f['line_number']}): `{f['line_preview']}`",
        ]

    lines += [
        "",
        "### 対処手順",
        "1. `git log --all --full-history -- <該当ファイル>` で漏洩範囲を確認",
        "2. 検出されたシークレットを即座に無効化・ローテーション",
        "3. `git filter-repo` または BFG Repo Cleaner でコミット履歴から削除",
        "4. force push 後、関係者に通知",
    ]

    return "\n".join(lines)
```

## 7.6 AIセキュリティレビューの限界と必要な補完策

AIによるセキュリティレビューには明確な限界がある。

**AIが検出できない脆弱性**

- **実行時のみ判明する問題**: レースコンディション（TOCTOU）は静的解析では完全に検出できない
- **設定ファイルの問題**: `nginx.conf` の誤設定、CSPヘッダーの欠落はdiffだけでは判断困難
- **依存関係の脆弱性**: `requests==2.20.0` の既知CVEはAIがリアルタイムの脆弱性DBを持たないため検出できない
- **ビジネスロジックの欠陥**: 割引コードが無制限に使えるバグはビジネス要件を知らないと検出できない

**必要な補完策**

```
AIレビュー（Claude）
    ↓ 文脈を理解した脆弱性検出

+ SASTツール（Bandit / Semgrep）
    ↓ パターンベースの高速スキャン

+ 依存関係スキャン（pip-audit / Dependabot）
    ↓ 既知CVEの検出

+ 動的テスト（DAST: OWASP ZAP）
    ↓ 実行時脆弱性の検出

+ ペネトレーションテスト（定期実施）
    ↓ 総合的なセキュリティ評価
```

AIは「第2のレビュアー」であり、セキュリティの全責任を担うものではない。特にクリティカルなシステムでは人間のセキュリティエンジニアによるレビューを必ず実施すること。

## 🚨 本番落とし穴

### 落とし穴① シークレット検出の false positiveでノイズが増える

テストコードには `sk-test-xxx` のようなダミーキーが含まれることが多い。`tests/` ディレクトリや `fixtures/` ディレクトリのファイルはシークレットスキャン対象から除外するロジックを追加しておかないと、大量の false positive に悩まされる。

```python
def should_skip_file(filepath: str) -> bool:
    skip_patterns = ["tests/", "test_", "fixtures/", "mock_", ".example"]
    return any(p in filepath for p in skip_patterns)
```

### 落とし穴② セキュリティ指摘を全てblockingにする

全ての指摘をPRのブロック条件にすると、開発速度が大幅に低下する。重大度別にルールを設定し、`critical` のみをブロック、`high` は警告、`medium` 以下はコメントのみ、という3段階の対応を設計しておくことが重要だ。

## 実務Tips

**定期的な全コードスキャン:** PRレビューは新規変更のみを対象とするが、既存コードに潜む脆弱性は検出できない。月1回、全ファイルを対象にAIセキュリティスキャンを実行するバッチジョブを設定することを推奨する。

**脆弱性の修正PRを自動生成:** `critical` 脆弱性を検出した際に、Claude APIで修正コードを生成し、自動でPRを作成するシステムを構築した事例がある。`gh pr create` と組み合わせることで実現できる。

## まとめ

- BanditはパターンマッチングのためIDOR・ロジックバグ・SSRFを見逃す。AIは文脈を理解するためこれらを補完できる
- OWASP Top 10のチェックポイントをプロンプトに変換することでレビューの抜け漏れを防ぐ
- AIの True Positive率は82%。Banditとの組み合わせで91%まで向上する
- シークレット漏洩は正規表現スキャンで高速に検出し、AI呼び出し前にブロックする
- AIセキュリティレビューは多層防御の1層であり、DAST・ペネトレーションテストとの組み合わせが必須
