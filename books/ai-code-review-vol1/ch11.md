## 11.1 AI幻覚（Hallucination）の事例と検知

AIコードレビューを運用していると、「正しそうに見えるが間違っている指摘」に出会う。これが幻覚（Hallucination）だ。

### 幻覚パターン① 存在しない関数を安全と誤認

```python
# レビュー対象のコード
import sqlite3

def get_user_by_email(db: sqlite3.Connection, email: str) -> dict | None:
    cursor = db.execute(
        "SELECT * FROM users WHERE email = ?", (email,)
    )
    row = cursor.fetchone()
    return dict(row) if row else None
```

**AIの指摘（幻覚あり）:**
```
✅ このコードは安全です。sqlite3.execute()はパラメータ化クエリを使用しており、
SQLインジェクションの脆弱性はありません。また、sqlite3.sanitize_input()が
内部で呼ばれているため、追加のバリデーションは不要です。
```

**問題点:** `sqlite3.sanitize_input()` という関数は存在しない。AIが「安全であることを正当化するために」実在しない関数を創作した。

```python
# 幻覚を検知するバリデーション
import ast
import importlib
import inspect


def validate_function_references(
    code: str, ai_review: str
) -> list[str]:
    """AIレビューで言及された関数が実際に存在するか検証する."""
    # AIレビューから関数名を抽出（簡易実装）
    import re
    mentioned_functions = re.findall(r"`(\w+\.\w+\(\))`", ai_review)

    false_claims = []
    for func_ref in mentioned_functions:
        module_name, func_name = func_ref.rstrip("()").rsplit(".", 1)
        try:
            module = importlib.import_module(module_name)
            if not hasattr(module, func_name):
                false_claims.append(f"{func_ref} は存在しない関数です")
        except ImportError:
            pass  # サードパーティライブラリは検証スキップ

    return false_claims
```

### 幻覚検知のための実装パターン

```python
# hallucination_detector.py
from dataclasses import dataclass
from enum import Enum


class ConfidenceLevel(Enum):
    HIGH = "high"      # AIの指摘を信頼できる
    MEDIUM = "medium"  # 要確認
    LOW = "low"        # 人間の確認が必須


@dataclass
class ReviewFinding:
    """AIレビューの個別指摘."""
    severity: str
    message: str
    code_reference: str
    confidence: ConfidenceLevel = ConfidenceLevel.MEDIUM


def assess_confidence(finding_text: str) -> ConfidenceLevel:
    """
    AIの指摘文から信頼度を推定する.

    確定的な言い切りは幻覚リスクが高い。
    コードから直接読み取れる問題は信頼度が高い。
    """
    # 低信頼度のシグナル（確実性を装った表現）
    low_confidence_signals = [
        "内部で自動的に",
        "標準ライブラリが処理",
        "最新バージョンでは",
        "推奨されています",  # 根拠なき推奨
        "〜することはありません",  # 過剰な安心保証
    ]

    # 高信頼度のシグナル（コードから直接観察可能）
    high_confidence_signals = [
        "行",
        "変数",
        "関数名",
        "format string",
        "パラメータ化されていない",
        "ハードコード",
    ]

    text_lower = finding_text.lower()
    low_score = sum(1 for s in low_confidence_signals if s in finding_text)
    high_score = sum(1 for s in high_confidence_signals if s in finding_text)

    if low_score >= 2:
        return ConfidenceLevel.LOW
    elif high_score >= 2:
        return ConfidenceLevel.HIGH
    else:
        return ConfidenceLevel.MEDIUM
```

## 11.2 悪いプロンプトが生む誤指摘

プロンプトの設計ミスは大量の false positive（誤った指摘）を生み、開発者の信頼を損なう。

### 悪いプロンプト事例① 観点が漠然としすぎる

```python
# ❌ 悪いプロンプト
BAD_PROMPT = """
以下のコードをレビューして、問題があれば指摘してください。
問題点をすべてリストアップしてください。

```diff
{diff}
```
"""
```

**結果:** 87件のコメントが投稿された。スタイルの好みから本当のバグまで混在し、開発者は全コメントを無視するようになった。

```python
# ✅ 改善後のプロンプト
GOOD_PROMPT = """
あなたはPythonのセキュリティ専門レビュアーです。

## 役割（Role）
セキュリティバグと実行時エラーのみに集中するレビュアー

## タスク（Task）
以下のdiffをレビューし、**バグと脆弱性のみ**を指摘してください。
スタイル・命名・パフォーマンスは対象外です。

## フォーマット（Format）
```json
{
  "issues": [
    {
      "severity": "critical|high|medium",
      "line": 行番号,
      "problem": "問題の説明（50字以内）",
      "fix": "修正方法（コード付き）"
    }
  ],
  "issue_count": 指摘件数
}
```

## 制約（Constraints）
- 指摘は最大**5件**まで
- diffに存在するコードから直接確認できる問題のみ
- 「〜の可能性があります」という推測は禁止。証拠があるものだけ

```diff
{diff}
```
"""
```

### 悪いプロンプト事例② スコープを絞らないシステムプロンプト

| プロンプトの問題 | 症状 | 改善策 |
|----------------|------|--------|
| 全部指摘させる | 1PRに50件以上のコメント | 最大件数を明記（5〜10件） |
| 重大度なし | 軽微な指摘がブロックになる | severity フィールドを必須化 |
| 証拠を求めない | 幻覚的な指摘が混在 | 「diffから直接確認できるもの限定」 |
| 出力形式なし | パース不能な自然文 | JSON スキーマを明示 |
| チームルール不明 | 既存コードと矛盾する提案 | コーディング規約をコンテキストに含める |

### A/Bテストによるプロンプト改善フロー

```python
# prompt_ab_test.py
import json
from dataclasses import dataclass, field
from typing import Callable


@dataclass
class PromptEvalResult:
    prompt_name: str
    true_positives: int = 0   # 実際にバグだったAI指摘
    false_positives: int = 0  # バグでなかったAI指摘
    false_negatives: int = 0  # AIが見落としたバグ
    total_issues: int = 0

    @property
    def precision(self) -> float:
        """精度: 指摘のうち本当にバグだった割合."""
        if self.total_issues == 0:
            return 0.0
        return self.true_positives / self.total_issues

    @property
    def recall(self) -> float:
        """再現率: 実際のバグのうちAIが発見した割合."""
        actual_bugs = self.true_positives + self.false_negatives
        if actual_bugs == 0:
            return 0.0
        return self.true_positives / actual_bugs

    @property
    def f1_score(self) -> float:
        """F1スコア: 精度と再現率の調和平均."""
        p, r = self.precision, self.recall
        if p + r == 0:
            return 0.0
        return 2 * p * r / (p + r)


def track_review_outcome(
    ai_finding: dict,
    human_verdict: str,  # "true_positive" | "false_positive" | "wont_fix"
    results: PromptEvalResult,
) -> None:
    """人間のレビュー判定をトラッキングして精度を測定する."""
    results.total_issues += 1
    if human_verdict == "true_positive":
        results.true_positives += 1
    elif human_verdict == "false_positive":
        results.false_positives += 1
```

## 11.3 AIを盲信した失敗談

### 失敗談① 「AIがOKなら大丈夫」マインドの蔓延

あるチームは導入3ヶ月後、レビュアーが「AIが指摘なしと言ってるので承認」とコメントだけ残してApproveするケースが増えた。

その結果、ビジネスロジックに起因する決済バグが本番に入り、売上影響が発生した。

```python
# 問題のあったコードの模式図
def apply_discount(order_total: float, coupon_code: str) -> float:
    coupon = Coupon.get(coupon_code)
    if coupon and coupon.is_valid():
        return order_total * (1 - coupon.discount_rate)
    return order_total
    # ↑ AIは「クーポンロジックは正しい」と判定
    # 人間が見逃した問題: coupon.is_valid()がExpiry日のみチェックしており、
    # 1ユーザーが同一クーポンを無制限回数使えるバグがあった
```

**対策:** 「AIがOKと言ったPRは人間レビューをスキップできる」というルールを撤廃した。AIレビューはFirst Pass（自動チェック）であり、人間レビューは常に必須とした。

```python
# review_gate.py — 人間確認ゲートの実装
from enum import Enum


class ReviewGate(Enum):
    """PRがマージ可能になる条件."""
    AI_ONLY = "ai_only"             # AIのみ（推奨しない）
    AI_REQUIRED = "ai_required"     # AIは必須、人間はオプション
    HUMAN_REQUIRED = "human_required"  # 人間は必須（AIはオプション）
    BOTH_REQUIRED = "both_required" # 両方必須（推奨）


# GitHub Actions で BOTH_REQUIRED を強制する設定例
MERGE_POLICY = {
    "default": ReviewGate.BOTH_REQUIRED,
    "docs_only": ReviewGate.AI_REQUIRED,  # ドキュメントのみのPR
    "hotfix": ReviewGate.HUMAN_REQUIRED,  # 緊急修正（AIを待てない場合）
}
```

### 失敗談② セキュリティ指摘を全てブロックにしたら開発が止まった

`critical` と `high` の指摘が1件でもあれば自動でブロックする設定を入れたところ、1日に30件以上のブロックが発生した。多くは false positive だったが、解除のプロセスが面倒なため開発速度が70%低下した。

```python
# ❌ 問題の設定
def should_block_pr(findings: list[dict]) -> bool:
    return any(f["severity"] in ("critical", "high") for f in findings)


# ✅ 改善後: 信頼スコア付きの判断
def should_block_pr_v2(findings: list[dict], confidence_threshold: float = 0.8) -> bool:
    """
    信頼スコアが閾値を超える場合のみブロックする.

    信頼スコアは過去の human_verdict データから学習する。
    """
    high_confidence_critical = [
        f for f in findings
        if f["severity"] == "critical"
        and f.get("confidence_score", 0.5) >= confidence_threshold
    ]
    return len(high_confidence_critical) > 0
```

### 失敗談③ 1つのモデルへの依存でコスト爆発

最初にSonnetをすべてのPRに使い始めたところ、1ヶ月で$180のAPIコストが発生した（予算$30の6倍）。

```python
# ✅ モデル選択の最適化
def select_review_model(diff: str, changed_files: list[str]) -> str:
    """
    変更の性質に応じてモデルを選択する.

    - セキュリティに関わるファイル → Sonnet（高精度）
    - 小さなdiff → Haiku（低コスト）
    - 大きなdiff → Haiku でサマリ後 Sonnet で深掘り
    """
    SECURITY_SENSITIVE_PATTERNS = [
        "auth", "login", "password", "token", "payment", "admin"
    ]

    is_security_critical = any(
        pattern in f.lower()
        for f in changed_files
        for pattern in SECURITY_SENSITIVE_PATTERNS
    )

    diff_lines = len(diff.split("\n"))

    if is_security_critical:
        return "claude-sonnet-4-6"
    elif diff_lines < 100:
        return "claude-haiku-4-5-20251001"
    else:
        return "claude-haiku-4-5-20251001"  # 大きいdiffもHaikuで初期分類
```

## 11.4 緩和戦略のまとめ

### 信頼スコアシステムの設計

```python
# trust_score.py
import json
from collections import defaultdict
from pathlib import Path


class TrustScoreTracker:
    """
    AIの指摘カテゴリ別に信頼スコアを追跡する.

    human_verdict の記録から、カテゴリ別の正解率を計算する。
    False Positive率が高いカテゴリは自動的に信頼スコアが下がる。
    """

    def __init__(self, data_path: str = "trust_scores.json"):
        self.data_path = Path(data_path)
        self._data: dict = self._load()

    def _load(self) -> dict:
        if self.data_path.exists():
            return json.loads(self.data_path.read_text())
        return defaultdict(lambda: {"tp": 0, "fp": 0})

    def record(self, category: str, is_true_positive: bool) -> None:
        """人間の判定を記録する."""
        if category not in self._data:
            self._data[category] = {"tp": 0, "fp": 0}
        if is_true_positive:
            self._data[category]["tp"] += 1
        else:
            self._data[category]["fp"] += 1
        self.data_path.write_text(json.dumps(self._data, indent=2))

    def get_score(self, category: str) -> float:
        """
        カテゴリの信頼スコアを返す（0.0〜1.0）.

        データが少ない場合は保守的な中間値（0.5）を返す。
        """
        d = self._data.get(category, {"tp": 0, "fp": 0})
        total = d["tp"] + d["fp"]
        if total < 10:  # データ不足
            return 0.5
        return d["tp"] / total
```

### 人間確認ゲートのチェックリスト

```
AIコードレビューの健全な運用チェックリスト:

【プロンプト設計】
  ✅ 指摘件数の上限を設定している（推奨: 最大10件）
  ✅ 観点を明確に絞っている（バグ+セキュリティのみ、など）
  ✅ 出力フォーマット（JSON）を強制している
  ✅ 証拠主義を明記（diffから直接確認できるもののみ）
  ✅ チームのコーディング規約をコンテキストに含めている

【運用ルール】
  ✅ AIが「問題なし」でも人間レビューは必須
  ✅ ビジネスロジックに関わるコードは必ず人間がレビュー
  ✅ Critical指摘のブロックは信頼スコアが0.8以上の場合のみ
  ✅ 月次でFalse Positive率を集計し、プロンプトを改善している

【コスト管理】
  ✅ ファイルの種類によってモデルを使い分けている
  ✅ 月次コストアラートを設定している
  ✅ ドキュメントのみのPRはAIレビューをスキップしている

【信頼管理】
  ✅ AIの指摘に対するhuman_verdictを記録している
  ✅ カテゴリ別のFalse Positive率を可視化している
  ✅ 信頼スコアが低いカテゴリはプロンプトを見直している
```

## まとめ

- **幻覚は必ず発生する**: AIは存在しない関数を「安全の根拠」として引用することがある。コードから直接読み取れる事実のみを根拠とするプロンプト設計が不可欠だ
- **プロンプトの粗さが誤指摘を生む**: 観点・件数・フォーマットの3つを制約しないと、大量の false positive でレビュアーの信頼を失う
- **AIはFirst Passであり最終判断者ではない**: 「AIがOKなら承認」というルールは組織として禁止する必要がある
- **信頼スコアは学習する**: human_verdict を蓄積することで、どのカテゴリの指摘を信頼できるかが分かってくる
- **コストは設計で制御できる**: モデル選択・スキャン対象・プロンプトの精度を最適化すれば、月数百円で運用できる
