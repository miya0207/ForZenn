# 第1章: AIと人間のコードレビュー — 根本的な違い

:::message
**この章で学べること**
- AIと人間がコードをどう「読む」のかの本質的な違い
- AIが得意/不得意な領域を具体的に理解する
- 「AIに全部任せる」でも「AIは使えない」でもない第三の道
:::

## 1.1 二つのレビューは「何を見ているか」が根本的に違う

あなたのチームに、二人のレビュアーが新しく加わったとする。

**レビュアーA（AI）**は、コードを受け取った瞬間から解析を始める。構文ツリーを走査し、データフローを追い、既知のパターンライブラリと照合する。1000行のコードを3秒で読み終え、発見した問題のリストを返す。疲れを知らず、機嫌も悪くならない。

**レビュアーB（人間の上級エンジニア）**は、まずPRの背景を読む。Jiraのチケット、Slack上の議論、先週のアーキテクチャ決定を思い出しながら、コードを読む。「なぜここでこの実装を選んだのか」を考える。コードが正しいかどうかより、*コードがここに存在すべきか*を考える。

同じ「コードレビュー」という言葉で呼ばれているが、二人がやっていることは根本的に異なる。

### AIのレビューは「パターンマッチング」

AIのコードレビューは、本質的には**高度なパターンマッチング**だ。

```python
# AIが発見しやすいパターンの例

# 1. N+1クエリ（パターンとして認識しやすい）
for user in users:
    posts = db.query(Post).filter(Post.user_id == user.id).all()  # N+1!

# 2. SQLインジェクション（既知の脆弱性パターン）
query = f"SELECT * FROM users WHERE name = '{name}'"  # 危険

# 3. 型エラーリスク（静的解析的な検出）
def process(items: list[str]) -> int:
    return sum(items)  # str を int として加算 → TypeError

# 4. 未使用変数（構文解析で検出）
def calculate(x: int, y: int) -> int:
    result = x + y
    temp = x * y  # 未使用
    return result
```

これらはすべて「コードの構造」から検出できる問題だ。AIは訓練データに含まれる無数のコードパターンを学習しており、類似したパターンを見つけることが得意だ。

### 人間のレビューは「文脈理解」

一方、人間のレビューは**文脈理解と価値判断**だ。

```python
# 人間が気づきやすいが、AIが見落としやすいパターン

# ケース1: 正しいが、チームの方針と合わない実装
class UserService:
    def __init__(self):
        self.db = Database()  # DIを使わずハードコード
        # → AIは「動く」と判断するが、チームはDIを方針としている

# ケース2: 将来の変更で壊れる設計
STATUS_CODES = {
    "active": 1,
    "inactive": 2,
}
# → 今は動くが、「suspended」を後で追加したとき、
#   この辞書を参照しているコードが20箇所あると壊滅的になる。
#   人間のレビュアーは「Enumを使うべき」と指摘する。

# ケース3: ビジネスルールの問題
def apply_discount(price: float, user_type: str) -> float:
    if user_type == "premium":
        return price * 0.8
    return price
    # → コードは正しいが、「プレミアムユーザーの20%引きは今月末で終わる」
    #   という話を知っている人間だけが、ハードコードの危険性に気づく
```

## 1.2 実測比較：同じコードを人間とAIがレビューした結果

下記のコードに対して、筆者がAI（Claude）と経験5年のエンジニアにそれぞれレビューを依頼した実験の結果を示す。

```python
# レビュー対象コード（実際に使ったもの）
import sqlite3
from datetime import datetime

class ArticleRepository:
    def __init__(self, db_path: str):
        self.conn = sqlite3.connect(db_path)  # 接続を使い回す

    def save(self, title: str, content: str, tags: list):
        cursor = self.conn.cursor()
        # タグをカンマ区切りで保存
        tags_str = ",".join(tags)
        cursor.execute(
            f"INSERT INTO articles (title, content, tags, created_at) "
            f"VALUES ('{title}', '{content}', '{tags_str}', '{datetime.now()}')"
        )
        self.conn.commit()

    def find_by_tag(self, tag: str) -> list:
        cursor = self.conn.cursor()
        # タグ検索（部分一致）
        result = cursor.execute(
            f"SELECT * FROM articles WHERE tags LIKE '%{tag}%'"
        ).fetchall()
        return result

    def get_recent(self, limit: int = 10) -> list:
        return self.conn.execute(
            f"SELECT * FROM articles ORDER BY created_at DESC LIMIT {limit}"
        ).fetchall()
```

### AIのレビュー結果（Claude API、2秒）

```
🔴 重大: SQLインジェクション脆弱性
  - save(): title, content, tags_str が直接クエリに埋め込まれています
  - find_by_tag(): tag パラメータが検索クエリに直接埋め込まれています
  - 修正: プレースホルダ (?) を使用してください

🔴 重大: SQLインジェクション（同上）
  - get_recent(): limit パラメータも整数ですが、パラメータ化が推奨です

🟡 警告: データベース接続の管理
  - sqlite3.connect() をコンストラクタで呼び出しており、接続が閉じられません
  - コンテキストマネージャ (with 文) またはデストラクタの実装を推奨します

🟡 警告: タグの設計
  - タグをカンマ区切りの文字列として保存しています
  - クエリの %LIKE% 検索は部分一致の誤検知が発生します
    例: "python" で検索すると "pythonic" にもヒットします

🔵 提案: 型アノテーション
  - find_by_tag の戻り値型が list で不明確です
  - list[tuple] または専用のデータクラスを推奨します
```

### 人間エンジニアのレビュー結果（15分）

```
SQLインジェクションは分かった。それより気になるのが設計の問題。

1. このクラス、どこで使うの？
   Webアプリなら、接続をコンストラクタで持つのは危険。
   マルチスレッドのリクエストが同時に来たとき、同じ接続を
   複数スレッドで共有することになる（スレッドセーフじゃない）。

2. タグのスキーマ設計が将来の問題を作っている。
   タグ検索、タグ数のカウント、タグ一覧取得…
   将来絶対やりたくなる機能が、このスキーマでは全部苦痛になる。
   article_tags テーブルを作って正規化してほしい。

3. get_recent() がタプルのリストを返しているが、
   呼び出し側はインデックス0がtitle、2がcontent...とか
   マジックナンバーで参照することになる。
   dataclassかnamedtupleにして。

4. テストどうするの？
   sqlite3.connect() に依存してるからモックできない。
   インターフェースを切って、DIで差し替えられるようにして。
```

### 比較から見えること

| 観点 | AI | 人間 |
|------|-----|------|
| SQLインジェクション発見 | ✅ 2秒 | ✅ 15分 |
| スレッドセーフティ問題 | ❌ 未指摘 | ✅ 指摘 |
| スキーマ設計の問題 | △ 部分的 | ✅ 完全指摘 |
| テスタビリティ問題 | ❌ 未指摘 | ✅ 指摘 |
| 型の改善提案 | ✅ 指摘 | ✅ より具体的 |
| レビュー時間 | 2秒 | 15分 |

## 1.3 AIが本質的に苦手なこと

### 「なぜ」が分からない

AIはコードの「何」は分かるが「なぜ」は分からない。

```python
# このコードに対してAIは何と言うか？
RETRY_COUNT = 3  # ← なぜ3なの？

def send_notification(user_id: int, message: str) -> bool:
    for i in range(RETRY_COUNT):
        try:
            result = smtp_client.send(user_id, message)
            return True
        except SMTPException:
            time.sleep(2 ** i)  # 指数バックオフ
    return False
```

AIは「`RETRY_COUNT`はマジックナンバーです。定数として定義してください」と言うかもしれない（すでに定数だが）。
しかし人間のシニアエンジニアは聞く：「なぜ3回？SLAは何秒？この指数バックオフの最大待ち時間（4秒）はタイムアウト設定と合っている？」

### ビジネスルールの妥当性

```python
def calculate_invoice(base_price: float, discount_rate: float) -> float:
    """
    請求金額を計算する。
    discount_rate: 0.0〜1.0（0.2 = 20%引き）
    """
    if discount_rate > 1.0:
        raise ValueError("割引率は1.0以下にしてください")

    discounted = base_price * (1 - discount_rate)
    tax = discounted * 0.10  # 消費税10%
    return discounted + tax
```

AIはこのコードを「正しい」と評価する。構文的にも論理的にも問題がない。

しかし人間は質問する：「割引率が負数（マイナス割引＝値上げ）は許容していいの？B2Bの特定顧客との契約で、消費税が免税のケースは？`discount_rate=0.999`で1円の請求が来たとき、会計システムは処理できる？」

これらはビジネスドメインの知識がないと判断できない問いだ。

### チームの文脈と慣習

```python
# このコードを見てAIは何と言うか

def process_user_data(user: dict) -> None:
    name = user["name"]
    email = user["email"]
    # ...処理...
```

AIは「`user: dict`より`user: User`（dataclass）を使うべき」と言うかもしれない。
しかし、チームが「外部APIのレスポンスを加工するレイヤーでは、あえてdictを使う」という決定をしているなら、この指摘は的外れだ。

AIはチームのSlackを読んでいないし、先月の設計議論を知らない。

## 1.4 人間が本質的に苦手なこと（正直に言う）

### 疲労と見落とし

研究によると、コードレビューの集中力は**30〜60分が限界**だ。

```python
# このパターン、5回目に見たときに気づけるか？
for i in range(len(users)):  # range(len())アンチパターン
    print(users[i].name)
```

1回目のPRなら指摘できる。しかし1日に10個のPRをレビューした後の10個目のPRで、疲弊したレビュアーがこれを見落とすことは珍しくない。

AIは疲れない。1000個目のPRでも同じ精度で指摘する。

### 感情とバイアス

人間は無意識に、書いた人によってレビューの厳しさが変わる。

- 上司が書いたコード → 甘くなる
- 苦手な同僚が書いたコード → 厳しくなる
- 自分が以前書いたコードと同じパターン → 見落とす

AIにはこのバイアスがない（ただし、訓練データのバイアスは別問題として存在する）。

### スケール問題

10人のチームで、全員が毎日2〜3個のPRを出したとする。1日25〜30個のPRに対して人間がレビューをすると、待ち時間が発生し、レビュー品質が下がり、「レビューしてない」PRがマージされ始める。

これは規模の問題であり、人間の努力でどうにかなる問題ではない。

## 1.5 最適な役割分担とは何か

本書全体を通じて主張したいことを、ここで先に述べておく。

```
AIが担うべき役割:
  ✅ 構文・パターンの問題検出（N+1、SQLインジェクション、型エラー）
  ✅ セキュリティの基本チェック（OWASP Top 10）
  ✅ スタイルの一貫性確認（Lintで拾えないもの）
  ✅ 24時間・均一品質での第一チェック
  ✅ レビュアーの疲弊防止（自明な問題を先に除去）

人間が担うべき役割:
  ✅ アーキテクチャ適合性の判断
  ✅ ビジネスルールの妥当性確認
  ✅ 設計の将来拡張性レビュー
  ✅ チームの文脈・慣習との整合性
  ✅ テスタビリティ・保守性の総合判断
  ✅ 最終的な「マージして良いか」の責任

やってはいけないこと:
  ❌ AIの指摘を全て受け入れる（コンテキストが欠けている場合がある）
  ❌ AIが「OK」と言ったから人間レビューを省く
  ❌ AIを使わずに人間が全てのPRをレビューしようとする（スケールしない）
```

## 🚨 本番落とし穴

### 落とし穴①：「AIが合格したから安全」という誤解

AIが「SQLインジェクションなし」と判断したPRに、実は認証バイパスの脆弱性があった——これは現実に起きる。

AIはチェックリストに基づいて検査するが、**チェックリストにない攻撃は検出できない**。セキュリティは多層防御が必要であり、AIは一層目に過ぎない。

### 落とし穴②：AIの指摘が多すぎてノイズになる

AIに「全部チェックして」と頼むと、100コメントが帰ってくることがある。開発者は100コメントを見て思考停止する。

**解決策：重要度でフィルタリングする。**`🔴 重大 / 🟡 警告 / 🔵 提案`の3段階で分類し、重大以外はサマリー形式にする（第5章で実装する）。

### 落とし穴③：AIが間違ったことを断言する

Claudeは間違えることがある。しかも確信を持った口調で。

```
Claude: "このコードにはバグがあります。xxメソッドはスレッドセーフではないため..."
実際:  Python 3.10以降ではこの問題は修正済みだった
```

AIの指摘は**提案として扱い、必ず開発者が判断する**。「AIが言ったから」はレビューを通す理由にならない。

## 実務Tips

**Tip 1: AIレビューは「第0レビュー」として位置づける**
AIの指摘を修正してからレビュアーにPRを出す運用にすると、人間レビュアーが本質的な設計議論に集中できる。

**Tip 2: AIの指摘ログを蓄積して傾向分析する**
チームのコードで頻繁に指摘されるパターンは、教育カリキュラムや静的解析ルールに昇格させると根本解決になる。

**Tip 3: 「AIが指摘しなかった」を記録する**
バグが本番で見つかったとき「AIは何と言っていたか」を確認する習慣をつける。AIの盲点が蓄積的に分かってくる。

## まとめ

本章の要点を整理する：

- **AIのレビューはパターンマッチング**：既知の問題を高速・網羅的に検出するが、「なぜ」の文脈は理解できない
- **人間のレビューは文脈理解**：ビジネスルール・チームの慣習・将来の変更を踏まえた価値判断
- **最適解は役割分担**：AIが第一チェック → 人間が設計・文脈レビュー
- **どちらも完璧ではない**：AIは疲れないが盲点がある。人間は文脈を読めるが疲れる

次章では、AIが「具体的に何を、どのように検出するか」を、実際のコードサンプルと検出率のデータで示す。
