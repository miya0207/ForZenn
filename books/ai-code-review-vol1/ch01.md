    def get_user(self, username: str):
        # ユーザー名でDBを検索
        query = f"SELECT * FROM users WHERE username = '{username}'"
        return self.conn.execute(query).fetchone()

    def list_active_users(self, users=[]):
        # アクティブユーザー一覧を返す

            users.append(user)
        return users

    def update_password(self, user_id: int, new_password: str):
        # パスワードを更新する
        self.conn.execute(
            f"UPDATE users SET password = '{new_password}' WHERE id = {user_id}"
        )

## ステップ3：AIのレビュー結果

実際にClaude Haikuが返した結果（数秒で出力）：

```
🔴 重大: SQLインジェクション脆弱性（get_user メソッド）

【問題】
username が直接クエリ文字列に埋め込まれています。
攻撃者が "admin' OR '1'='1" を入力すると全ユーザーが返ります。

【修正】
def get_user(self, username: str):
    return self.conn.execute(
        "SELECT * FROM users WHERE username = ?",
        (username,)
    ).fetchone()

---

🔴 重大: SQLインジェクション脆弱性（update_password メソッド）

【問題】
new_password と user_id がクエリに直接埋め込まれています。

【修正】
self.conn.execute(
    "UPDATE users SET password = ? WHERE id = ?",
    (new_password, user_id)
)
self.conn.commit()  # コミット漏れも修正

---

🟡 警告: ミュータブルデフォルト引数（list_active_users メソッド）

【問題】
def list_active_users(self, users=[]) の users=[] は、
全ての呼び出しで同じリストを共有します。
2回目の呼び出しでは前回のデータが残ります。

【修正】
def list_active_users(self, users=None):
    if users is None:
        users = []
    ...

---

🟡 警告: トランザクション管理の漏れ（update_password）

self.conn.commit() が呼ばれていないため、
パスワード変更がDBに保存されません。

---

🔵 提案: データベース接続管理

sqlite3.connect() をコンストラクタで固定しているため、
テスト時にモックが困難です。
コンテキストマネージャまたは依存性注入の使用を検討してください。
```

## ステップ4：なぜAIがこれを検出できたか

AIはこの4つの問題を、以下の仕組みで検出した：

**SQLインジェクション** → `f"...{変数}..."` というfストリング + SQL文字列の組み合わせパターンが既知の脆弱性パターンと一致。数百万のコードサンプルでこのパターンを学習済み。

**ミュータブルデフォルト引数** → `def func(arg=[])` の構文パターンを認識。Pythonの挙動として「全呼び出しで同じオブジェクトを参照する」ことを学習済み。

**コミット漏れ** → `execute()` の後に `commit()` がない場合のパターンを認識。

**DB接続管理** → コンストラクタでのDB接続固定というアンチパターンを認識し、テスタビリティの問題として指摘。

## 人間なら何を追加で指摘するか

AIの指摘は全て正確だった。しかし、人間のシニアエンジニアはさらに以下を指摘する：

```
1. なぜこのクラスがDB接続を直接持つのか？
   → UserRepositoryとして分離し、サービス層とDBアクセスを分けるべき

2. パスワードを平文で保存しようとしている
   → bcryptなどのハッシュ化ライブラリを使うべき（AIはこれを見落とした）

3. このコードにテストはあるか？
   → 現状の設計ではユニットテストが書けない
```

AIは「コードの構造的な問題」を見つけた。
人間は「設計上の問題」と「ビジネスリスク」を見つけた。

これが本書全体のテーマだ。
