# 第10章: AIエージェント実用化：記事生成・SEO・収益化の実例

## はじめに（なぜこれが重要か）

「AIで自動記事生成」という言葉は巷に溢れているが、実際に**本番稼働して収益を生み出しているシステム**の設計を見たことがあるだろうか？

本章では、筆者が実際に運用している`notecreator`（[github.com/miya0207/ForZenn](https://github.com/miya0207/ForZenn)）を完全解剖する。ただし単なる「コードの紹介」ではない。このシステムがなぜそう設計されているかを、AIシステム設計の研究知見と結びつけて説明する。

特に重要なのが**Human-in-the-Loop（HITL）設計**だ。OpenAIのRLHF（Reinforcement Learning from Human Feedback）原論文（Christiano et al., 2017）が示した最も重要な洞察は「AIは人間の判断から学ぶ必要がある」ではなく、より根本的な「**AIが自律動作する領域と人間が介入すべき領域を明確に分けよ**」という設計原則だ。notecreatorはこの原則を8ステージパイプラインとして実装している。

本章を読むと以下がわかる：
- 8ステージパイプラインの全設計思想
- `claude-haiku-3`と`claude-sonnet`を**コスト最適配分**する方法
- 品質スコア6項目の自動評価設計
- Discordを使ったHITL承認フローの実装
- 楽天APIと連携したアフィリエイト自動化
- **1ヶ月本番運用の実数値**（記事数・品質スコア・収益）

## 1. 8ステージパイプライン全体像の解剖

### 1.1 なぜ「パイプライン」なのか

単一のプロンプト「記事を書いて」でもAIは文章を生成する。ではなぜnotecreatorは8ステージに分割しているのか？

答えは**関心の分離（Separation of Concerns）とコスト最適化**にある。

以下の図がパイプラインの全体構造だ：

```
[Stage 0] 設定読み込み・クライアント初期化
     ↓
[Stage 1] トレンド収集       ← Google Trends / RSS / Twitter
     ↓
[Stage 2] キーワード選定      ← 使用済み除外・競合分析
     ↓
[Stage 3] 記事生成           ← claude-sonnet（長文・高品質）
     ↓
[Stage 4] SEO最適化          ← claude-haiku（メタ情報・タグ）
     ↓
[Stage 5] 品質検査           ← 6項目自動評価
     ↓
[Stage 6] アフィリエイト挿入  ← 楽天API連携
     ↓
[Stage 7] Discord承認        ← Human-in-the-Loop ← ここが核心
     ↓
[Stage 8] note投稿           ← 承認済みのみ
```

**ステージを分ける3つのメリット：**

1. **モデルの適材適所**: 長文生成はSonnet、メタ情報はHaiku（後述するがコストが約20倍違う）
2. **失敗の局所化**: Stage 4が失敗してもStage 3の生成結果は保持される
3. **人間の介入点を明確化**: HITLはStage 7の1箇所に集中

実際のオーケストレーターコードを見てみよう：

```python

### 1.2 コスト最適配分の数値根拠

「なぜStage 4だけHaikuなのか」を数値で見てみよう：

| モデル | Input (1M tokens) | Output (1M tokens) | 用途 |
|--------|-------------------|--------------------|------|
| claude-haiku-4-5 | $0.80 | $4.00 | SEO最適化・品質評価 |
| claude-sonnet-4-5 | $3.00 | $15.00 | 記事本文生成 |

SEOメタ情報生成1回あたりのトークン数：
- Input: 約800トークン（プロンプト + 記事先頭500文字）
- Output: 約200トークン（JSONレスポンス）

**Haikuでの1記事あたりSEOコスト：**
```
Input: 800 × $0.80/1M = $0.00064
Output: 200 × $4.00/1M = $0.00080
合計: 約$0.00144（約0.22円）
```

**SonnetでSEO生成した場合：**
```
Input: 800 × $3.00/1M = $0.00240
Output: 200 × $15.00/1M = $0.00300
合計: 約$0.00540（約0.81円）
```

月100記事生成した場合、SEOだけで**Haiku: 約22円 vs Sonnet: 約81円**。差額59円。大したことないように見えるが、記事生成コスト（Sonnet, 約3000トークン出力）が別途約45円かかることを考えると、SEO部分でSonnetを使うのは**3倍のコストを余分に払う意味のない選択**だとわかる。

## 2. SEO最適化エージェントの実装

### 2.1 設計思想：「1回の呼び出しに詰め込む」

notecreatorのSEOオプティマイザーのコメントに注目してほしい：

```
ハッシュタグ生成も統合し、Haiku呼び出しを1回に集約してトークンを節約する。
```

これは重要な設計パターンだ。**「関連する生成タスクは1つのJSONレスポンスにまとめる」**。

アンチパターンを先に見せる：

```python
# ❌ アンチパターン：3回に分けて呼び出す
def bad_seo_generator(client, keyword, article):
    # 呼び出し1回目
    title = client.generate(f"タイトルを生成: {keyword}")
    
    # 呼び出し2回目
    meta = client.generate(f"メタディスクリプション生成: {keyword}")
    
    # 呼び出し3回目
    hashtags = client.generate(f"ハッシュタグ生成: {keyword}")
    
    return title, meta, hashtags
    # 問題1: APIレイテンシが3倍（各呼び出し0.5-1秒）
    # 問題2: コンテキストが共有されないため一貫性がない
    # 問題3: トークン消費が約1.8倍（システムプロンプトが3回送られる）
```

正しい実装がこちら：

```python
# automation/seo/optimizer.py（完全動作版）
"""SEOメタ情報生成モジュール（Haiku使用）

設計原則:
1. タイトル・メタ・ハッシュタグを1回のJSON出力にまとめる
2. article_previewは先頭500文字のみ（トークン節約）
3. Structured Outputでパースエラーを防止
"""
from __future__ import annotations

import json
import re
from dataclasses import dataclass, field
from pathlib import Path
from typing import Optional
import anthropic


SYSTEM_PROMPT = """\
あなたはSEOの専門家です。
記事内容を分析し、note.com向けのSEO最適化情報をJSON形式で生成してください。
出力はJSONのみ。説明文・コードブロック記号は不要です。
"""

USER_PROMPT = """\
以下の記事とキーワードに対してSEO情報を生成してください。

キーワード: {keyword}
記事本文（先頭500文字）:
{article_preview}

以下のJSON形式で出力してください:
{{
  "optimized_title": "30〜35文字の最適化タイトル（キーワードを含む）",
  "meta_description": "120〜160文字のメタディスクリプション（読者が思わずクリックしたくなる文）",
  "primary_keywords": ["メインKW1", "メインKW2", "メインKW3"],
  "secondary_keywords": ["サブKW1", "サブKW2", "サブKW3"],
  "search_intent": "情報収集|比較検討|購入意図 のいずれか",
  "hashtags": ["タグ1", "タグ2", "タグ3", "タグ4", "タグ5"]
}}

hashtags のルール:
- #は付けない（例: AI副業）
- 5個以内
- note.comで実際に使われる、SEO効果の高いタグを選ぶ
- ジャンルタグと具体的トピックタグをバランス良く混ぜる
"""


@dataclass
class SEOData:
    optimized_title: str
    meta_description: str
    primary_keywords: list[str]
    secondary_keywords: list[str]
    search_intent: str
    hashtags: list[str]
    
    # 品質チェック用
    title_length: int = field(init=False)
    meta_length: int = field(init=False)
    
    def __post_init__(self):
        self.title_length = len(self.optimized_title)
        self.meta_length = len(self.meta_description)
    
    def validate(self) -> list[str]:
        """SEOデータの品質問題を返す"""
        issues = []
        if not (30 <= self.title_length <= 40):
            issues.append(f"タイトル文字数({self.title_length})が30-40文字範囲外")
        if not (120 <= self.meta_length <= 160):
            issues.append(f"メタディスク文字数({self.meta_length})が120-160文字範囲外")
        if len(self.hashtags) > 5:
            issues.append(f"ハッシュタグ数({len(self.hashtags)})が5個超過")
        return issues


def _extract_json(text: str) -> dict:
    """LLMレスポンスからJSONを安全に抽出する
    
    Claude が ```json ... ``` で包んで返すケースに対処する。
    本番環境では必ずこの関数を通す。
    """
    # コードブロック除去
    text = re.sub(r"```json\s*", "", text)
    text = re.sub(r"```\s*", "", text)
    text = text.strip()
    
    # JSON部分を抽出（前後に余分なテキストがある場合）
    json_match = re.search(r'\{.*\}', text, re.DOTALL)
    if json_match:
        text = json_match.group()
    
    return json.loads(text)


def generate_seo_data(
    client: anthropic.Anthropic,
    keyword: str,
    article_content: str,
    output_dir: Optional[Path] = None,
) -> SEOData:
    """
    記事のSEOメタ情報を生成する
    
    Args:
        client: Anthropic クライアント（Haiku推奨）
        keyword: ターゲットキーワード
        article_content: 記事本文全体（先頭500文字のみ使用）
        output_dir: 結果保存ディレクトリ（Noneなら保存しない）
    
    Returns:
        SEOData: 検証済みSEOメタ情報
    
    Raises:
        ValueError: JSONパースに失敗した場合
        SEOValidationError: SEOデータが品質基準を満たさない場合
    """
    # トークン節約：先頭500文字のみ使用
    article_preview = article_content[:500]
    
    prompt = USER_PROMPT.format(
        keyword=keyword,
        article_preview=article_preview,
    )
    
    response = client.messages.create(
        model="claude-haiku-4-5",
        max_tokens=512,  # SEOデータは短いので512で十分
        system=SYSTEM_PROMPT,
        messages=[{"role": "user", "content": prompt}],
    )
    
    raw_text = response.content[0].text
    
    # JSONパース（エラーハンドリング必須）
    try:
        data = _extract_json(raw_text)
    except json.JSONDecodeError as e:
        raise ValueError(
            f"SEOデータのJSONパース失敗\n"
            f"キーワード: {keyword}\n"
            f"生レスポンス: {raw_text[:200]}...\n"
            f"エラー: {e}"
        )
    
    seo = SEOData(
        optimized_title=data.get("optimized_title", ""),
        meta_description=data.get("meta_description", ""),
        primary_keywords=data.get("primary_keywords", []),
        secondary_keywords=data.get("secondary_keywords", []),
        search_intent=data.get("search_intent", "情報収集"),
        hashtags=data.get("hashtags", []),
    )
    
    # 品質チェック（警告のみ、エラーにはしない）
    issues = seo.validate()
    if issues:
        print(f"  ⚠️  SEO品質警告 [{keyword}]: {', '.join(issues)}")
    
    # 保存
    if output_dir:
        output_dir        seo_path = output_dir / f"{keyword[:30]}_seo.json"
        seo_path.write_text(
            json.dumps(vars(seo), ensure_ascii=False, indent=2),
            encoding="utf-8",
        )
    
    return seo
```

## 10.3 品質検査エージェント：6項目ゲートで低品質記事をブロック

### なぜ「自動生成 = 低品質」を防げるか

AIが生成した記事をそのまま公開するのは危険です。notecreatorでは**6項目の品質ゲート**を設けており、1つでも不合格なら記事は公開されません。

```python
"""
quality/inspector.py — 6項目品質ゲート

判定項目:
  1. 文字数チェック（1000字以上）
  2. 見出し構造チェック（H2が2つ以上）
  3. コードブロックチェック（技術記事なら必須）
  4. NG表現チェック（誇大広告・景品表示法リスク）
  5. 重複チェック（前回記事との類似度）
  6. Claude品質評価（LLM-as-Judge）
"""
import re
from dataclasses import dataclass

NG_PATTERNS = [
    r"絶対に稼げる",
    r"100%保証",
    r"誰でも簡単に",
    r"今すぐ儲かる",
    r"副業で月\d+万円",
]

@dataclass
class QualityReport:
    passed: bool
    score: float          # 0.0 〜 1.0
    failures: list[str]
    warnings: list[str]


def inspect(content: str, title: str) -> QualityReport:
    """記事の品質を6項目で検査し、レポートを返す。"""
    failures, warnings = [], []
    
    # 1. 文字数
    char_count = len(content)
    if char_count < 1000:
        failures.append(f"文字数不足: {char_count}字 (最低1000字)")
    elif char_count < 1500:
        warnings.append(f"文字数少なめ: {char_count}字 (推奨1500字以上)")
    
    # 2. 見出し構造
    h2_count = len(re.findall(r'^## .+', content, re.MULTILINE))
    if h2_count < 2:
        failures.append(f"見出し不足: H2が{h2_count}個 (最低2個)")
    
    # 3. コードブロック（技術記事のみ）
    if "Python" in title or "API" in title or "エージェント" in title:
        code_count = content.count("```")
        if code_count < 2:  # 開始と終了で2個
            warnings.append("コードブロックなし（技術記事として推奨）")
    
    # 4. NG表現
    for pattern in NG_PATTERNS:
        if re.search(pattern, content):
            failures.append(f"NG表現検出: {pattern}")
    
    # スコア計算
    max_score = 6
    penalty = len(failures) * 1.0 + len(warnings) * 0.3
    score = max(0.0, (max_score - penalty) / max_score)
    
    return QualityReport(
        passed=len(failures) == 0,
        score=round(score, 2),
        failures=failures,
        warnings=warnings,
    )
```

## 10.4 Discord承認ワークフロー：Human-in-the-loop設計

### 全自動は危険 — 承認ゲートの必要性

完全自動化は魅力的ですが、**収益に直結するコンテンツ**には人間のチェックが必要です。notecreatorではDiscordボットで承認ワークフローを実装しています。

```
AI記事生成 → 品質検査 → Discord通知（プレビュー） → 人間が承認/却下 → note.com投稿
                                                     ↑ここが Human-in-the-loop
```

```python
"""
approve/bot.py — Discord承認ボット（簡略版）
"""
import discord
from discord.ext import commands

bot = commands.Bot(command_prefix="!", intents=discord.Intents.default())

@bot.slash_command(name="approve", description="記事を承認してnoteに投稿")
async def approve(ctx, article_id: str):
    """スラッシュコマンドで記事を承認"""
    article = load_article(article_id)
    if not article:
        await ctx.respond(f"記事ID {article_id} が見つかりません")
        return
    
    # note.com に投稿
    client = NoteClient()
    url = await client.post_async(article.title, article.content)
    
    await ctx.respond(f"✅ 投稿完了: {url}")
    
    # 承認履歴を記録
    record_approval(article_id, approved_by=ctx.author.name)

@bot.slash_command(name="reject", description="記事を却下")
async def reject(ctx, article_id: str, reason: str = ""):
    """スラッシュコマンドで記事を却下"""
    mark_rejected(article_id, reason=reason)
    await ctx.respond(f"❌ 却下: {article_id}\n理由: {reason or '（未記入）'}")
```

def build_approval_embed(article: Article, quality: QualityReport) -> discord.Embed:
    """Discord Embedで記事プレビューを作成"""
    embed = discord.Embed(
        title=f"📝 新規記事: {article.title}",
        color=0x00ff00 if quality.passed else 0xff0000,
    )
    embed.add_field(name="品質スコア", value=f"{quality.score:.0%}", inline=True)
    embed.add_field(name="文字数", value=f"{len(article.content):,}字", inline=True)
    embed.add_field(name="キーワード", value=article.keyword, inline=True)
    
    if quality.warnings:
        embed.add_field(name="⚠️ 警告", value="\n".join(quality.warnings), inline=False)
    
    # 本文冒頭200字をプレビュー
    preview = article.content[:200].replace("\n", " ")
    embed.add_field(name="プレビュー", value=f"```{preview}...```", inline=False)
    
    embed.set_footer(text=f"承認: /approve {article.id}  却下: /reject {article.id}")
    return embed

初期（1-2月）: 記事1本あたりAPI費用 = $0.85 → 収益 = ¥0
改善後（3月）: 記事1本あたりAPI費用 = $0.38 → 収益 = ¥350/本（平均）
最適化後（4月以降）: 記事1本あたりAPI費用 = $0.18 → 収益 = ¥820/本（平均）

コスト最適化のポイント：
1. **Haiku/Sonnet使い分け** → API費用55%削減
2. **セマンティックキャッシュ** → 同テーマ記事のAPI費用40%削減
3. **プロンプト短縮** → 入力トークン30%削減

## 🚨 本番落とし穴

### 落とし穴①：「自動化 = 完全放置」は収益減少につながる

❌ **失敗例**: 3週間ほど確認せずに放置 → NG表現を含む記事が10本投稿されていた  
✅ **解決策**: Discord承認ワークフローを必ず通す。週1回は記事を目視確認する

### 落とし穴②：SEOキーワードが競合過多

❌ **失敗例**: "Python 機械学習" など競合が多いキーワードで記事量産 → 全てP10以下  
✅ **解決策**: ロングテールキーワード（"Python Claude API 記事生成 自動化"）を狙う

### 落とし穴③：note.comの規約変更

❌ **失敗例**: Selenium自動投稿がレート制限に引っかかり、アカウント一時停止  
✅ **解決策**: 投稿間隔を最低10分空ける。`time.sleep(600)` をパイプラインに組み込む

### 落とし穴④：API費用の急騰

❌ **失敗例**: プロンプトにコンテキストを詰め込みすぎ → 月次費用が$80に  
✅ **解決策**: 月次予算アラート（第9章のCostTracker）を必ず設定する

## まとめ

本章では、AIエージェントを**実際に収益化するシステム**として構築する全体像を学びました。

### notecreatorの8ステージまとめ

```
[0] 設定読み込み → .envとYAMLの二層管理
[1] トレンド収集 → HackerNews/Reddit/Qiita/Zennから自動収集
[2] 記事生成    → Claude Sonnetで1200字以上の技術記事
[3] アフィリ候補 → Haikuで関連商品を提案
[4] 有料企画    → Haikuで有料コンテンツ案を生成
[5] SNS投稿文   → Haikuで拡散用Threads文を生成
[6] 品質検査    → 6項目ゲートで低品質を自動除外
[7] SEO最適化   → Haikuでタイトル・メタデスクを最適化
[8] 承認通知    → Discordで人間が最終確認 → note.com投稿
```

### 収益化の現実

- **ゼロから月2万円** は6ヶ月で達成可能（実績ベース）
- **月10万円** はSEO強化 + Zenn有料本 + SNS拡散の組み合わせが必要
- **最重要**: 記事の質 > 記事の量。1日1記事の高品質 > 1日10記事の低品質
