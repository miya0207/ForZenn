# Pythonで作るClaudeエージェント：プロンプト設計から本番運用まで

**notecreator実装を題材に、ReAct・ツール設計・コスト管理を徹底解説**

---

## まず、これを読んでほしい

本番でAIエージェントを動かすと、こんなことが起きる。

```
月曜日 09:12 — エージェントが記事を生成し始める
月曜日 09:14 — Web検索ツールが404を返す
月曜日 09:14 — エージェントがリトライする
月曜日 09:15 — リトライする
月曜日 09:16 — リトライする
...
月曜日 09:45 — APIコスト $47.82 を消費して無限ループ終了
```

私はこれを経験した。**本番で。**

開発環境では完璧に動いていた。単体テストも通っていた。でも本番のトラフィックと実データが合わさった瞬間、エージェントは盛大に壊れた。

**「LLMを呼ぶコードを書ける」と「LLMエージェントを設計できる」は、全く別のスキルだ。**

この本は、その違いを体系的に教える本だ。

---

## この本を開いた時点で、あなたはこれを経験しているはずだ

:::message alert
**本番の現実：よくある失敗パターン**

- APIコストが想定の10倍になった（コンテキスト管理の設計ミス）
- エージェントが無限ループに入り、手動で殺すまで止まらなかった（停止条件の設計ミス）
- プロンプトを少し変えたら、別のタスクの品質が崩壊した（プロンプトリグレッション）
- 本番障害が起きたが「どのステップで何が起きたか」が全くわからなかった（可観測性の欠如）
- 同じ入力でも週によって品質が30%ブレる（確率的出力のテスト設計ミス）
:::

これらはすべて、設計の問題だ。

Claudeが賢くなってもこれらは解決しない。エージェントの信頼性は、**LLMの周囲のシステム設計**で決まる。

これは絵に描いた餅ではない。本書の実装例は、**notecreator**という実際に稼働しているプロダクションシステムをベースにしている。

notecreatorは毎朝9時に起動し、トレンドを収集し、記事を生成し、品質検査を通過したものだけをnote.comに投稿する。6ヶ月の稼働実績があり、月¥24,720の収益を83%の利益率で生み出している。

本書のコードはそのシステムから抽出したものだ。

| | 学術書 | チュートリアル | **本書** |
|--|--------|--------------|---------|
| 研究論文の知識 | ✅ 深い | ❌ 浅い | ✅ 引用・解説あり |
| 動くコード | ❌ 疑似コードのみ | ✅ 動く | ✅ 本番品質 |
| 現場ノウハウ | ❌ ほぼない | ❌ トイプロブレム止まり | ✅ 失敗談つき |
| コスト設計 | ❌ | ❌ | ✅ 実測値あり |
| 信頼性設計 | ❌ | ❌ | ✅ 本番実績あり |

**LangChainを使わない理由**

本書はあえてLangChainなどのフレームワークを使わない。理由は一つ：

> **フレームワークは本番で裏切る。内部実装を知らない人間は、障害発生時に詰む。**

Anthropic公式SDKだけを使い、すべての仕組みをゼロから組み立てる。LangChainの中身を理解してから使うことで、初めてフレームワークを「使いこなす」ことができる。

## この本の対象読者

**✅ こんな人に刺さる**

- Python実務経験2〜5年。APIやDB操作は普通に書ける
- `anthropic.messages.create()` を呼んだことはある。でも「設計として正しいAIシステム」の作り方がわからない
- LangChainを使ってみたが、「中で何が起きているか」がわからなくて不安
- プロダクトにAIを組み込む仕事が来そう（または来ている）で急いでキャッチアップしたい
- Anthropicの料金体系を見て「本番で使ったらいくらかかるの...？」と不安になった

**❌ この本が向かない読者**

- Python入門者（`def`関数やクラスが理解できていない段階）
- 学術研究として最先端LLMを研究したい人
- ノーコード・ローコードでAIを使いたい人
